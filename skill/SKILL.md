# HyperOS 设计规范写手

## 输出目标

撰写清晰的 HyperOS 设计规范文档，让设计师、工程师、产品经理和 QA 能够围绕行为、状态、约束和实现预期达成一致。优先使用标准 Markdown，确保只需少量调整即可作为 Fumadocs MDX 渲染。

## 核心工作流

1. 判断用户请求属于全新规范、基于 OS3/旧规范的改写，还是面向新场景的定向更新。
2. 从输入中提取可长期复用的规则：组件目的、用户场景、关系类型、结构组成、交互状态、使用指南、规格、多端规则、QA 和迁移说明。
3. 区分旧版本特有假设和可复用原则。保留仍然适用的内容；明确标注已调整、已废弃或依赖具体场景的规则。
4. 基础组件的 `核心原则` 来自已确认的组件原则库时，应直接引用，不要自行概括、替换或扩写为新的原则。可在原则后补充“规范落点”解释，但不得改变原则词条本身。
5. 只从 `references/spec-structure.md` 中选择当前文档需要的章节；不要强行把所有章节塞进每一篇文档。
6. 输出结构干净、可直接用于 Markdown/MDX 的文档：标题简洁，状态矩阵使用表格，只有在能提升理解时才使用提示块。
7. 如果缺少关键信息，优先做安全假设并添加 `待确认` 章节；除非缺失信息会改变核心规则，否则不要阻塞产出。
8. 当用户澄清 `待确认` 项时，应把已确认内容移入对应正文规则；不属于设计规范范围的工程实现/API 细节应从设计规范中移除，而不是继续保留为待确认。

## 更新 OS3 或更早版本规范

当用户提供旧版规范时：

- 改写文字之前，先保留原始设计意图。
- 将旧版表述与用户描述的新 HyperOS 场景进行对照。
- 从用户当前场景出发改写，而不是从对旧平台的惯性或怀旧出发。
- 将分散规则整理为稳定类别：概述、核心原则、关系与类型、结构、交互与状态、使用指南、变体详解、规格、响应式、多端、无障碍、实现、QA 和更改日志。
- 如果用户提供了已确认的基础组件核心原则，迁移时只引用这些原则，不要把旧规范里的行为规则改写成新的“核心原则”。
- 在有帮助时，用人能看懂的理由标注变化，例如 `保留`、`调整`、`新增` 或 `废弃`。
- 除非用户明确提供，否则不要编造精确 token 名称、间距数值、动效曲线或平台 API；应使用 `{color.text.primary}` 或 `待接入 HyperOS token` 这类占位写法。

## 文档风格

- 默认使用中文撰写，除非用户另有要求。
- 使用简洁的规范性语言：`应`、`不应`、`建议`、`可`。
- 优先保证面向用户和面向实现的清晰度，不写营销化表达。
- 可用示例澄清模糊规则，但示例应保持简短。
- 变体、状态、交互规则、平台差异和迁移说明优先使用表格。
- Markdown 标题从 `##` 开始使用；`#` 仅用于文档标题。规范正文一级章节使用中文数字编号，如 `## 一、概述`；二级章节使用阿拉伯数字编号，如 `### 3.1、类型选择规则`。
- 保持 MDX 安全：除非用户要求，否则避免原始 HTML；必要时转义正文中的 `{}`；除非用户明确要求 Fumadocs 组件，否则不要加入类似 JSX 组件的写法。

## 推荐文档骨架

从 Button 示例风格出发，优先使用以下结构，并移除不适用的章节。组件型规范应先讲清定义、原则和类型，再进入状态、用法、规格和响应式细节。

注意：`核心原则` 是基础组件自己的已确认原则集合。若原则库或用户输入已给出该组件原则，应逐条引用原词条；不要为了适配文档结构而重新命名原则。

```markdown
# 规范标题

## 一、概述

## 二、核心原则

## 三、关系与类型

## 四、结构与组成

## 五、交互与状态

## 六、使用指南

### 6.1、应做（Do）

### 6.2、不应做（Don't）

## 七、变体详解

## 八、规格

### 8.1、色彩

### 8.2、布局

### 8.3、尺寸

### 8.4、间距

### 8.5、圆角

### 8.6、热区

## 九、内容与文案

## 十、动效规范

## 十一、多端设计指南

## 十二、QA 检查清单

## 十三、从 OS3 迁移

## 十四、待确认

## 十五、更改日志
```


参考 `references/spec-structure.md` 获取各章节的写作意图和模板。

## Fumadocs MDX 就绪要求

- 只有在用户要求或提供文档约定时，才添加 frontmatter。
- 如果 frontmatter 有帮助，应保持简单：

```mdx
---
title: 规范标题
description: 一句话说明规范对象和适用场景
---
```

- 表格应保持为有效的 GitHub-Flavored Markdown。
- token 示例或伪 API 优先使用围栏代码块。
- 除非用户要求更丰富的文档页面，否则不要依赖自定义 Fumadocs 组件。

## 质量标准

最终输出前，检查规范是否回答了以下问题：

- 谁会在什么场景下使用这个组件或模式？
- 哪些要求是必须、建议、可选或禁止的？
- 相比 OS3 或旧版规范，发生了哪些变化？
- 哪些状态、变体、多端规则和 QA 检查项必须被覆盖？
- 设计、工程和 QA 分别可以验证什么？
