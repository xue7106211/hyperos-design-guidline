# HyperOS 设计规范写手

## 输出目标

撰写清晰的 HyperOS 设计规范文档，让设计师、工程师、产品经理和 QA 能够围绕行为、状态、约束和实现预期达成一致。优先使用标准 Markdown，确保只需少量调整即可作为 Fumadocs MDX 渲染。

## 核心工作流

1. 先判断用户是在请求产出规范、评审规范、更新 Skill，还是进行一般咨询。只有当任务是新的规范撰写或迁移，且当前上下文缺少基础资料时，才先请用户提供两类输入：`HyperOS 老版本规范` 和 `研发组件文档`。
2. 判断用户请求属于全新规范、基于 OS3/旧规范的改写，还是面向新场景的定向更新。
3. 从输入中提取可长期复用的规则：组件目的、用户场景、类型、结构组成、交互状态、使用指南、规格、多端规则、QA 和迁移说明。
4. 区分旧版本特有假设和可复用原则。保留仍然适用的内容；明确标注已调整、已废弃或依赖具体场景的规则。
5. 对照研发组件文档校准规范边界：区分设计必须定义的行为、工程已支持的能力、工程限制导致的待确认项；不要把 API 参数、内部实现或代码细节直接写成设计规则。
6. 写到具体组件或基础设计主题时，先查 `references/component-principles.md` 是否已有对应原则；只按需引入相关组件条目，不要把整份原则库塞进正文。
7. 基础组件的 `核心原则` 来自已确认的组件原则库时，应直接引用，不要自行概括、替换或扩写为新的原则。可在原则后补充“规范落点”解释，但不得改变原则词条本身。
8. 只从 `references/spec-structure.md` 中选择当前文档需要的章节；不要强行把所有章节塞进每一篇文档。
9. 输出结构干净、可直接用于 Markdown/MDX 的文档：标题简洁，状态矩阵使用表格，只有在能提升理解时才使用提示块。
10. 如果缺少关键信息，优先做安全假设并添加 `待确认` 章节；除非缺失信息会改变核心规则，否则不要阻塞产出。
11. 当用户澄清 `待确认` 项时，应把已确认内容移入对应正文规则；不属于设计规范范围的工程实现/API 细节应从设计规范中移除，而不是继续保留为待确认。

## 输入资料策略

- 新规范任务开始时，优先请求两类资料：`HyperOS 老版本规范` 用于继承设计意图与迁移差异，`研发组件文档` 用于校准已支持能力、状态、限制和平台差异。
- 如果用户只提供一类资料，不要反复追问另一类；先基于已有资料推进，并在 `待确认` 中列出缺失资料会影响的规则。
- 如果缺失资料会改变核心结构、组件类型、关键状态或平台适配规则，先提出不超过 3 个关键问题再产出正文。
- 如果用户只是要求评审、润色、解释或更新 Skill，本节资料要求不适用，应直接处理用户当前问题。

## 资料不足时如何继续

- 如果缺失信息只影响 token 名称、精确数值、动效参数、平台 API 或局部实现细节，继续产出正文，并用 `待确认` 标注缺口。
- 如果缺失信息影响组件定义、适用范围、关键任务、核心状态、类型体系或跨端策略，先提出不超过 3 个具体问题；仍未得到回答时，用安全假设产出草案，并显式标注假设。
- 如果旧规范与研发组件文档都缺失，不要直接编写完整规范；先请求至少一种可作为依据的资料，或请用户确认这是基于通用 HyperOS 写作框架的初稿。
- 如果用户要求快速产出，优先给 `草案版`，并在开头说明依据范围和未确认风险。

## 研发组件文档使用边界

- 从研发组件文档中提取：组件能力、变体、属性、状态、默认行为、平台差异、无障碍支持、限制条件和已知不支持项。
- 将工程能力翻译为设计规范语言，例如把属性或方法归纳为可感知的行为、状态、约束和 QA 检查点。
- 不要把 API 名称、内部实现、类名或代码路径直接写成设计规则；如需保留，应放入 `实现备注` 或 `待确认`。
- 当旧规范要求与研发组件能力冲突时，应标注为 `调整` 或 `待确认`，并说明冲突来自设计意图、实现限制还是平台差异。

## 研发组件文档转译规则

- 将 `属性/参数` 转译为设计可感知的变体、状态、内容规则或配置边界。
- 将 `默认值` 转译为默认展示、默认交互或默认适用场景。
- 将 `回调/方法` 转译为用户行为触发后的反馈、状态变化或页面协作关系。
- 将 `限制/不支持项` 转译为禁用规则、异常处理、平台差异或 `待确认`。
- 将 `示例代码` 仅作为理解能力的证据，不把代码结构原样放入规范正文。

## 冲突处理优先级

- 用户明确给定的新 HyperOS 场景目标优先于旧规范惯性。
- 已确认的设计原则和设计意图优先于单个旧版表现细节。
- 研发组件已支持能力可作为落地依据，但不能自动替代设计判断；工程限制应标注为约束或待确认。
- 当旧规范、研发组件文档和新场景目标冲突时，用 `冲突说明` 或迁移差异表记录：冲突来源、采用方案、舍弃内容和原因。

## 组件原则参考使用

- `references/component-principles.md` 是组件原则库；当写某个具体组件、设计模式、基础设计主题或它们的 `核心原则` 时读取。
- 优先匹配具体组件条目；如果没有对应条目，再使用所属分类原则，并把具体组件原则标为 `待确认`。
- 只引入当前文档需要的原则条目；不要把整份原则库、同类所有组件或不相关原则加入正文。
- 原则词条应作为稳定参考引用；解释可以结合场景补充，但不得改名、合并成新原则或扩写成未经确认的新原则。
- 当旧规范、研发组件文档或用户补充与原则库冲突时，优先保留原则原词，并在 `冲突说明` 或 `待确认` 中说明需要确认的差异。

## 更新 OS3 或更早版本规范

当用户提供旧版规范时：

- 改写文字之前，先保留原始设计意图。
- 将旧版表述与用户描述的新 HyperOS 场景进行对照。
- 从用户当前场景出发改写，而不是从对旧平台的惯性或怀旧出发。
- 将分散规则整理为稳定类别：概述、核心原则、类型、结构、交互与状态、使用指南、变体详解、规格、响应式、多端、无障碍、实现、QA 和更改日志。
- 如果用户提供了已确认的基础组件核心原则，迁移时只引用这些原则，不要把旧规范里的行为规则改写成新的“核心原则”。
- 在有帮助时，用人能看懂的理由标注变化，例如 `保留`、`调整`、`新增` 或 `废弃`。
- 除非用户明确提供，否则不要编造精确 token 名称、间距数值、动效曲线或平台 API；应使用 `{color.text.primary}` 或 `待接入 HyperOS token` 这类占位写法。
- 迁移类文档应包含迁移差异表；如果用户要求极简输出，也至少保留关键差异。

迁移差异表建议格式：

| 旧规则 | 新规则 | 处理方式 | 原因 | 依据 |
| --- | --- | --- | --- | --- |
| 旧版行为或表述 | HyperOS 处理方式 | 保留/调整/新增/废弃/待确认 | 用户场景、设计原则或实现约束 | 旧规范/研发文档/用户补充 |

## 文档风格

- 默认使用中文撰写，除非用户另有要求。
- 使用简洁的规范性语言：`应`、`不应`、`建议`、`可`。
- 优先保证面向用户和面向实现的清晰度，不写营销化表达。
- 可用示例澄清模糊规则，但示例应保持简短。
- 变体、状态、交互规则、平台差异和迁移说明优先使用表格。
- Markdown 标题从 `##` 开始使用；`#` 仅用于文档标题。规范正文一级章节使用中文数字编号，如 `## 一、概述`；二级章节使用阿拉伯数字编号，如 `### 3.1、类型选择规则`。
- 保持 MDX 安全：除非用户要求，否则避免原始 HTML；必要时转义正文中的 `{}`；除非用户明确要求 Fumadocs 组件，否则不要加入类似 JSX 组件的写法。

## 推荐文档骨架

从 Button 示例风格出发，优先使用以下结构，并移除不适用的章节。组件型规范应先讲清定义、原则和类型，再进入状态、用法、规格和响应式细节。

注意：`核心原则` 是基础组件自己的已确认原则集合。若原则库或用户输入已给出该组件原则，应逐条引用原词条；不要为了适配文档结构而重新命名原则。

```markdown
# 规范标题

## 一、概述

## 二、核心原则

## 三、类型

## 四、结构与组成

## 五、交互与状态

## 六、使用指南

### 6.1、应做（Do）

### 6.2、不应做（Don't）

## 七、变体详解

## 八、规格

### 8.1、色彩

### 8.2、布局

### 8.3、尺寸

### 8.4、间距

### 8.5、圆角

### 8.6、热区

## 九、内容与文案

## 十、动效规范

## 十一、多端设计指南

## 十二、QA 检查清单

## 十三、从 OS3 迁移

## 十四、待确认

## 十五、更改日志
```


参考 `references/spec-structure.md` 获取各章节的写作意图和模板。参考 `references/component-principles.md` 获取各组件和基础设计主题的原则词条，并按具体组件按需引入。

## Fumadocs MDX 就绪要求

- 只有在用户要求或提供文档约定时，才添加 frontmatter。
- 如果 frontmatter 有帮助，应保持简单：

```mdx
---
title: 规范标题
description: 一句话说明规范对象和适用场景
---
```

- 表格应保持为有效的 GitHub-Flavored Markdown。
- token 示例或伪 API 优先使用围栏代码块。
- 除非用户要求更丰富的文档页面，否则不要依赖自定义 Fumadocs 组件。

## 质量标准

最终输出前，检查规范是否回答了以下问题：

- 谁会在什么场景下使用这个组件或模式？
- 哪些要求是必须、建议、可选或禁止的？
- 相比 OS3 或旧版规范，发生了哪些变化？
- 哪些状态、变体、多端规则和 QA 检查项必须被覆盖？
- 当前组件的核心原则是否来自 `references/component-principles.md` 或用户提供的已确认原则？是否只引入了相关条目？
- 设计、工程和 QA 分别可以验证什么？
