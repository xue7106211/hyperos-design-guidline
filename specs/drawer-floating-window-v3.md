# 抽屉浮窗 Drawer Floating Window V3

## 一、概述

抽屉浮窗是从屏幕边缘进入的容器，用来承载临时任务、轻量页面、辅助配置，也可以和背景内容并行工作。

同一个抽屉浮窗可以在「抽屉形态」和「浮窗形态」之间切换。手机竖屏通常使用底部抽屉；大屏、折叠屏、平板和横屏下，可按断点和任务需要转为浮窗。

## 二、核心原则

- **非打扰性**：抽屉浮窗用于呈现补充信息、临时任务或并行配置，尽量不打断主任务。模态场景用遮罩和返回规则说明任务边界；非模态场景不要遮蔽或抢占背景操作。
- **层级与结构**：抽屉浮窗需要让用户看清它和背景页面、标题区、内容区、操作区之间的层级。容器高度、遮罩、阴影、圆角、标题和把手共同表达这层结构。
- **上下文相关**：抽屉浮窗的内容要和发起入口或背景页面有关联。打开、关闭、内部跳转、系统返回和大屏转化，都要保留这条来源线索。
- **连续性**：展开、收起、档位切换、抽屉/浮窗切换、屏幕旋转和分屏变化时，任务状态、内容位置和返回顺序要保持连续，避免突然跳变或状态丢失。
- **明确清晰**：抽屉浮窗需要说清当前任务、可操作区域、关闭方式、拖拽热区和状态反馈。标题、操作文案、空状态、错误状态、加载状态和手势反馈都要避免歧义。

## 三、背景关系与类型

抽屉浮窗不使用强调层级。它的类型判断主要看两件事：背景是否还能交互，以及当前窗口是否需要从底部抽屉转为浮窗。

### 3.1、背景关系

| 背景关系 | 适用类型 | 规则 |
| --- | --- | --- |
| 背景不可交互 | 模态抽屉窗口、模态抽屉页面 | 默认有遮罩。用户需要先处理抽屉内任务，关闭后回到发起页面。 |
| 背景可交互 | 非模态抽屉 | 不显示遮罩。背景内容可以继续预览或操作，抽屉高度变化可以影响背景布局。 |
| 视任务而定 | 大屏浮窗 | 用于横屏、折叠屏、平板或宽窗口。窗口高于页面内容，但宽度和位置按浮窗规则收住。 |

### 3.2、类型总览

| 类型 | 背景关系 | 用途 | 典型场景 | 关键约束 |
| --- | --- | --- | --- | --- |
| 模态抽屉窗口 | 背景不可交互 | 临时展示少量信息或操作项 | 操作菜单、少量配置、确认前选择 | 应有遮罩；背景内容不可操作；关闭后返回发起页面。 |
| 模态抽屉页面 | 背景不可交互 | 用较轻的方式承载原本的临时全屏页 | 新建闹钟、编辑条目、跨进程打开轻量页、搜索选择 | 可展开到较大档位；系统返回应退出抽屉页面或回退内部页面。 |
| 非模态抽屉 | 背景可交互 | 与背景内容并行预览、编辑或配置 | 日历日程联动、小部件编辑、AOD 预览配置 | 不应有遮罩；背景可继续操作；抽屉高度变化可影响背景内容布局。 |
| 大屏浮窗 | 视任务而定 | 抽屉在大屏或横屏下的转化形态 | 平板、折叠屏、横屏、宽窗口 | 不应简单铺满底部；应按浮窗规则控制宽度、位置。 |

### 3.3、形态策略

| 策略 | 规则 | 设计含义 |
| --- | --- | --- |
| 自动响应式 | 默认策略，根据容器宽高、断点和业务类型在抽屉与浮窗之间自动切换。 | 作为通用默认策略。 |
| 固定抽屉 | 始终使用底部抽屉形态。 | 用于依赖底部进入、窄屏优先，或暂未适配浮窗的场景。 |
| 固定浮窗 | 始终使用浮窗形态。 | 用于大屏优先、需要独立窗口感，或不适合贴底展示的任务。 |

抽屉切到浮窗时，任务含义、内部状态、返回顺序和内容结构保持一致；容器宽度、位置、把手和遮罩可以随形态调整。浮窗形态默认不显示拖动把手。只有需要拖动窗口或调整高度时才展示。

### 3.4、类型选择规则

| 任务关系 | 推荐类型 | 判断依据 |
| --- | --- | --- |
| 用户必须先处理浮层任务，背景仅作为来源 | 模态抽屉窗口 / 模态抽屉页面 | 背景不可交互，避免任务被打断。 |
| 用户需要边看背景边配置当前任务 | 非模态抽屉 | 背景内容需要预览或协作，不要用遮罩阻断。 |
| 内容是原全屏页的轻量调用 | 模态抽屉页面 | 保留页面结构，用更轻的方式打开。 |
| 宽屏下底部抽屉阅读跨度过长 | 大屏浮窗 | 控制窗口宽度，收住视觉焦点。 |

## 四、结构与组成


| 元素   | 必需    | 说明                                        |
| ---- | ----- | ----------------------------------------- |
| 容器   | 是     | 承载标题区与内容区；应具备独立层级、圆角、背景、阴影或等效分层表现。        |
| 遮罩   | 视类型而定 | 模态抽屉应提供遮罩并阻断背景交互；非模态抽屉不应提供遮罩。             |
| 滑动把手 | 可选    | 用于提示可拖拽、切换高度或关闭；当仅有一个固定档位且不可关闭时，可不展示。     |
| 标题区  | 可选    | 可包含标题、关闭、取消、确认或其他页面级操作；规则应继承标题栏，但不支持大字标题。 |
| 内容区  | 是     | 展示核心内容、表单、列表、宫格、选择器或配置项；底部按钮栏也算内容区。       |
| 操作区  | 可选    | 有主操作、确认、取消等底部操作时，要和内容滚动区域分清。              |




### 4.1、把手展示模式


| 模式         | 规则                       | 适用场景                                |
| ---------- | ------------------------ | ----------------------------------- |
| 占位模式       | 把手占据容器顶部一行空间，内容区从把手下方开始。 | 默认场景，内容顶部不需要贴边展示。                   |
| Overlay 模式 | 把手叠加在内容之上，不额外挤占内容区高度。    | 内容区顶部有图片、封面、ActionBar 或需要贴近面板边缘的视觉。 |


开启 Overlay 模式后，把手使用 overlay 适配样式。业务自定义把手时，需要继承基础 overlay 样式，避免丢失暗色、热区、对比度等默认适配。

## 五、交互与状态



### 5.1、状态矩阵


| 状态   | 触发条件             | 视觉变化                     | 行为规则                |
| ---- | ---------------- | ------------------------ | ------------------- |
| 默认   | 抽屉已打开且可交互        | 容器、标题区、内容区按默认层级展示        | 用户可操作内容或切换档位。       |
| 拖拽中  | 用户拖动把手、标题区或内容区顶部 | 容器跟随手势移动；背景遮罩或并行区域可按进度变化 | 释放后吸附到目标档位或关闭。      |
| 最大档位 | 抽屉已到 L 档或业务最大高度  | 顶部仍保留抽屉形态与安全间距           | 继续上滑仅回弹，不应进入无定义高度。  |
| 最小档位 | 抽屉已到收起态或业务最小高度   | 展示最小可识别内容或入口             | 继续下滑可关闭，除非业务定义不可关闭。 |
| 加载中  | 内容尚未完成加载         | 内容区展示加载反馈，框架保持稳定         | 加载完成后不要突然跳档。        |
| 空状态  | 内容为空或暂无结果        | 内容区展示空状态说明与可选行动          | 不要用关闭抽屉代替空状态说明。     |
| 错误状态 | 加载失败或权限不足        | 内容区展示错误信息与重试/引导          | 不要让用户停在空白容器里。       |
| 禁用关闭 | 任务必须完成或不可中断      | 关闭入口隐藏或禁用，并提供原因或替代操作     | 谨慎使用，避免把用户困住。       |




### 5.2、研发状态映射


| 设计语义         | BottomSheet 状态        | 说明                                            |
| ------------ | --------------------- | --------------------------------------------- |
| 收起态          | `STATE_COLLAPSED`     | 面板保留最小可见高度或入口；是否允许跳过由业务配置。                    |
| M 档 / 半展开参考态 | `STATE_HALF_EXPANDED` | 这个状态不是“显示控件自身高度的一半”，而是控件顶部距离窗口顶部约半个窗口高度。      |
| L 档 / 展开态    | `STATE_EXPANDED`      | 面板进入展开状态；当内容或配置达到 full height 时，表现为贴近顶部的统一状态。 |
| 关闭 / 隐藏      | `STATE_HIDDEN`        | 仅当业务允许隐藏或关闭时可进入。                              |


只有内容区自身高度达到 full height 阈值，或业务强制 full height 时，才启用 `STATE_HALF_EXPANDED`；否则跳过半展开态，避免面板脱离底部。

### 5.3、打开与关闭


| 行为   | 模态抽屉                            | 非模态抽屉                                    |
| ---- | ------------------------------- | ---------------------------------------- |
| 打开   | 从发起控件或底部边缘进入，背景进入遮罩与不可交互状态。     | 从底部边缘进入，背景保持可见且可交互。                      |
| 关闭按钮 | 可提供关闭、取消或返回入口。                  | 若抽屉中途被拉出或可被移除，应提供关闭入口；若抽屉始终存在且不可关闭，可不提供。 |
| 点击遮罩 | 默认存在遮罩，点击遮罩等同关闭操作。              | 不存在遮罩。                                   |
| 系统返回 | 应关闭抽屉并回到发起页面。                   | 按业务定义关闭抽屉或返回背景页面，避免不可预期。                 |
| 手势关闭 | 在最小档位继续下滑可关闭；高风险任务应要求显式确认或保存提示。 | 可关闭场景遵循同样规则；不可关闭场景不响应关闭手势。               |




### 5.4、档位切换


| 操作区域        | 上滑规则                                          | 下滑规则                                        |
| ----------- | --------------------------------------------- | ------------------------------------------- |
| 标题区 / 把手    | 未到最大档位时，短上滑切换到临近更大档位，长上滑切换到最大档位；已到最大档位时仅轻微回弹。 | 未到最小档位时，短下滑切换到临近更小档位，长下滑可关闭；已到最小档位时继续下滑可关闭。 |
| 内容区顶部或无滚动容器 | 优先扩展抽屉档位；无更大档位时再滚动内容或触发回弹。                    | 优先收缩抽屉档位；无更小档位时关闭抽屉。                        |
| 内容区中间       | 优先滚动内容，不要抢成抽屉高度变化。                            | 优先滚动内容回到顶部；到顶部后再触发抽屉收缩或关闭。                  |




### 5.5、拖动热区


| 热区              | 规则                                |
| --------------- | --------------------------------- |
| 把手              | 默认可作为整体拖动入口；浮窗形态默认不展示把手。          |
| 标题区 / ActionBar | 建议注册为拖动热区，尤其是 overlay 模式或固定标题栏场景。 |
| 内容区顶部 / 底部      | 仅当内容滚动到边界时，才可将纵向滑动转交给抽屉整体拖动。      |
| 普通内容区           | 不默认触发整体拖动，避免和列表、表单、横滑组件、地图等内容冲突。  |


拖动热区用位移阈值区分点击和拖拽：未超过阈值时，保留原视图的点击、长按；超过阈值后，抽屉接管并跟随手指移动。

## 六、使用指南



### 6.1、应做（Do）

- 临时任务需要保留背景上下文时，使用抽屉浮窗，不直接跳全屏页。
- 先判断背景是否还能操作，再选择模态或非模态。
- 每个业务入口都要定义默认档位、最大档位、最小档位，以及是否可关闭。
- 横屏、折叠屏和平板要判断继续用底部抽屉，还是转为浮窗；横屏可以继续使用底部抽屉。
- 内容滚动和抽屉拖拽之间要有明确的手势优先级。
- 未保存内容、错误、空状态和权限不足，都要给出反馈。



### 6.2、不应做（Don't）

- 同一页面不要同时打开多个抽屉浮窗，除非已经定义好层级和返回顺序。
- 不要把复杂全屏流程塞进抽屉。导航、返回和状态保存会变得不清楚。
- 非模态抽屉不要带遮罩；模态抽屉打开后，背景不要继续操作。
- 不要把 `STATE_HALF_EXPANDED` 理解为“展示内容的一半”。
- 普通内容区不要任意位置都触发整体拖动，避免和内容交互冲突。
- 如果为了 edge-to-edge 关闭系统避让，需要同时处理底部导航栏、小白条、IME 和安全区。



## 七、变体详解



### 7.1、模态抽屉窗口

用于临时展示少量信息或操作项，一般不放复杂内部导航。

#### 常见使用场景

- 操作菜单、轻量配置、确认前选择。
- 系统能力入口，如分享、图片选择、文件选择等封装能力。



#### 布局

- 默认使用底部抽屉形态。
- 内容较少时可自适应高度；内容超过 M 档容量时应进入 L 档或使用抽屉页面。
- 应提供遮罩并阻断背景交互。



#### 约束

- 关闭后应返回发起页面。
- 默认存在遮罩，点击遮罩等同关闭操作。



### 7.2、模态抽屉页面

用于以更轻量方式打开原本的临时全屏页，保留页面级标题区和内部导航能力。

#### 常见使用场景

- 新建、编辑、跨进程调用轻量页、搜索选择。



#### 布局

- 可展开到 L 档。
- 承载全屏页时，标题栏建议使用导航吸顶样式。
- 内部页面跳转时，抽屉框架、遮罩关系和把手规则应保持稳定。



#### 约束

- 系统返回先回退抽屉内部层级；没有内部层级时关闭抽屉。
- 不要在抽屉内再嵌套新的模态抽屉。



### 7.3、非模态抽屉

用于并行任务，背景可继续预览或操作，抽屉与背景形成协同关系。

#### 常见使用场景

- 日历和日程协同、小部件编辑、AOD 预览配置、背景预览同时编辑。



#### 布局

- 不展示遮罩。
- 抽屉高度变化可以影响背景内容布局；被遮挡的背景区域需要重排，或明确不可见。
- 单档且不可关闭时可隐藏把手。



#### 约束

- 抽屉中途被拉出，或允许移除时，要提供关闭入口。
- 背景和抽屉的焦点顺序要可预测。



### 7.4、大屏浮窗

用于横屏、折叠屏、平板或宽窗口下的转化形态。

#### 常见使用场景

- 手机横屏、折叠屏展开态、平板、应用分屏宽窗口。



#### 布局

- 宽度和位置按浮窗规则控制，不横向铺满。
- 业务保留抽屉形态时，空间足够可水平居中，宽度接近浮窗。



#### 约束

- 形态切换不改变任务语义、内部状态和返回顺序。
- 浮窗形态默认不显示拖动把手。



## 八、规格



### 8.1、色彩


| 属性   | Token / 规则                   | 备注                           |
| ---- | ---------------------------- | ---------------------------- |
| 容器背景 | 应与当前页面背景形成清晰层级，并支持浅色/深色模式。   | 待接入 `{color.surface.*}`。     |
| 遮罩   | 模态抽屉使用遮罩降低背景强调度；非模态抽屉不使用遮罩。 | 待接入 `{color.overlay.*}`。     |
| 标题文字 | 使用标题栏对应文字层级，不支持大字标题。         | 待接入 `{font.title.*}`。        |
| 把手   | 低强调、可识别，不应与标题或主操作竞争。         | 待接入 `{color.icon.tertiary}`。 |




### 8.2、布局


| 场景        | 规则                                           |
| --------- | -------------------------------------------- |
| 手机竖屏      | 默认从底部弹出，宽度贴合当前窗口宽度。                          |
| 手机横屏      | 允许继续使用底部抽屉；也可根据业务需要转化为浮窗。                    |
| 手机横屏但保持抽屉 | 空间足够时水平居中，并使用与浮窗一致或接近的宽度。                    |
| 折叠屏 / 平板  | 模态抽屉窗口与模态抽屉页面建议转化为浮窗；非模态抽屉可继续以半屏/分栏形态服务并行任务。 |
| 左右分屏      | 每个应用分屏区域可独立使用抽屉浮窗；不跨越当前应用窗口边界。               |
| 上下分屏      | 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。                 |




### 8.3、尺寸


| 档位    | 规则                                                  | 适用                  |
| ----- | --------------------------------------------------- | ------------------- |
| 收起态   | 半屏高度 = 半屏内标题区的高度。                                   | 非模态抽屉、少数可持续存在的工具面板。 |
| M 档   | 半屏高度 = 屏幕高度 * 70%。                                  | 多数默认半屏任务。           |
| L 档   | 半屏高度 = 屏幕高度 - 状态栏 - 间距。                             | 抽屉页面、复杂内容、长列表、表单编辑。 |
| 自适应高度 | 内容在 S-M 之间时，根据内容多少自适应高度，最低不低于 XX%；内容超过 M 时直接使用 L 档。 | 少量内容的模态抽屉窗口。        |
| 自定义档位 | 由业务定义默认高度、最大高度、最小高度与切换逻辑。                           | 非模态抽屉、需要背景预览的编辑工具。  |


高度变化应遵循有限档位，不支持无极悬停。若内容因大字号或多语言超过 M 档容量，建议直接进入 L 档。

### 8.4、间距

- L 档顶部应保留状态栏、安全区与顶部间距。
- 键盘弹出时，抽屉应避让键盘，保证当前输入项与主操作可见。
- 底部操作区应与内容滚动区域保持清晰边界。
- 具体间距待接入 HyperOS spacing token。



### 8.5、圆角


| 形态   | 规则                         |
| ---- | -------------------------- |
| 底部抽屉 | 顶部边缘需要有浮层容器的形态；底部贴合窗口或安全区。 |
| 浮窗   | 应使用四角圆角，并与浮窗体系保持一致。        |


圆角数值待接入 `{radius.*}`。

### 8.6、阴影 / 海拔

抽屉浮窗应表达高于页面内容的 Z 轴层级；非模态抽屉尤其需要清晰区分抽屉与背景内容。具体海拔待接入 `{elevation.*}`。

### 8.7、热区

- 把手、标题区或 ActionBar 可作为拖动热区。
- 关闭、取消、完成等操作应满足平台最小触控热区。
- 图标通常不随文本等比例放大，但触控热区不应缩小。



## 九、内容与文案

- 标题应说明当前任务，而不是重复发起按钮文案。
- 关闭、取消、完成、保存等操作文案应与任务结果一致；存在未保存内容时应明确提示。
- 空状态、错误状态和权限状态应放在内容区内处理，不应改变抽屉类型。
- 多语言长标题应遵从标题栏适配规则；业务不应依赖固定字数保证布局。



## 十、动效规范

- 打开与关闭应使用上下方向的进入/退出动效，来源方向与抽屉位置一致。
- 档位切换应吸附到明确高度，并给出自然回弹；不要停在未定义的中间高度。
- 模态抽屉打开时，遮罩透明度应与容器进入同步变化。
- 非模态抽屉调整高度时，背景内容若需要避让或重排，应与抽屉运动协调完成。
- 本规范不定义具体动画 token、时长、曲线和降级策略。



## 十一、多端设计指南



### 11.1、高度规则

- 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。
- 内容超过 M 档容量时，直接使用 L 档，不要无限增高。
- 内容区自身高度未达到 full height 阈值时，应跳过 `STATE_HALF_EXPANDED`。
- full height 状态下若希望内容区撑满剩余空间，应显式配置内容高度匹配父容器。



### 11.2、宽度规则

- 手机竖屏默认贴合当前窗口宽度。
- 横屏允许继续使用底部抽屉；具体宽度策略可按业务和后续断点规则确定。
- 折叠屏和平板断点与窗口最大宽度待单独补充；横屏允许继续使用底部抽屉。
- 分屏中不跨越当前应用窗口边界。



### 11.3、位置规则

- 底部抽屉从底部进入并吸附在底部安全区之上。
- 浮窗形态应按浮窗主题规则控制位置与大小。
- Edge-to-edge 场景如关闭系统自动避让，业务必须自行处理底部导航栏、小白条、IME 与安全区。



### 11.4、多项排布

- 底部操作如果存在多个按钮，应根据可用宽度和多语言长度决定横排或纵排。
- 同一界面在所有多语言下，操作区布局建议保持一致。



### 11.5、多端规则


| 终端       | 规则                                   |
| -------- | ------------------------------------ |
| 手机       | 竖屏优先使用底部抽屉；横屏允许继续使用底部抽屉，也可按业务转化为浮窗。  |
| 折叠屏      | 折叠窄屏可沿用手机规则；展开态应评估浮窗、分栏或非模态半屏。       |
| 平板       | 模态任务优先使用浮窗；并行任务可保留非模态半屏或分栏式抽屉。       |
| 分屏       | 每个分屏窗口独立计算宽高、位置和安全区；不跨窗口展示。          |
| 键鼠 / 触控板 | 需要支持焦点、键盘返回、点击 outside，以及滚轮/触控板滚动仲裁。 |




## 十二、QA 检查清单

- [ ] 模态抽屉打开后，背景不可点击、不可聚焦，系统返回可关闭或回退内部页面。
- [ ] 非模态抽屉打开后，背景仍可按预期预览或操作，且无遮罩。
- [ ] 标题区、内容区顶部、内容区中间的上滑/下滑优先级符合规范。
- [ ] 标题区或 ActionBar 已作为拖动热区接入，且点击与拖拽不会互相误触。
- [ ] M/L/收起态/自定义档位能正确吸附，不停留在未定义高度。
- [ ] `STATE_HALF_EXPANDED` 只在 full height 条件满足时出现，不会造成面板脱离底部。
- [ ] 长内容、大字号、多语言、空状态、错误状态不会破坏布局。
- [ ] 键盘弹出时，输入项和主操作仍可见且可操作。
- [ ] Edge-to-edge 场景已处理底部导航栏、小白条、IME 与安全区。
- [ ] 手机竖屏、横屏、折叠屏、平板、分屏下的形态转化符合业务定义。
- [ ] 深色模式与浅色模式下，容器层级、遮罩和文字可读性一致。



## 十三、从 OS3 迁移


| 旧规范规则                          | V3 处理 | 原因                                                     |
| ------------------------------ | ----- | ------------------------------------------------------ |
| 模态抽屉用于临时任务，背景有遮罩且不可交互          | 保留    | 仍是区分临时任务与并行任务的核心规则。                                    |
| 非模态抽屉用于并行任务，背景可交互              | 保留并强化 | HyperOS 多任务、大屏与预览编辑场景更依赖此规则。                           |
| 抽屉高度遵循低、中、高三个尺寸，不支持无极悬停        | 调整    | V3 使用收起态、M、L、自适应和业务自定义档位，仍保留有限档位吸附。                    |
| 横屏暂不支持，强制竖屏                    | 调整    | V3 明确横屏可以继续使用底部抽屉，也可以按业务转为浮窗。                          |
| 折叠屏、PAD 默认为浮窗，小部件特殊适配          | 保留并泛化 | V3 将它整理为大屏转化规则；非模态抽屉可按并行任务保留半屏形态。                      |
| 标题栏规则与全屏页面一致，但不支持大字标题          | 保留    | 维持页面级导航一致性，同时避免抽屉标题区占用过高。                              |
| 非模态单档且不可关闭时可隐藏把手               | 保留    | 避免把手暗示不存在的拖拽或关闭能力。                                     |
| 点击遮罩不可关闭窗口                     | 调整    | V3 明确模态抽屉默认有遮罩，点击遮罩就是关闭。                               |
| 研发 BottomSheet 支持抽屉与浮窗形态切换     | 新增    | V3 保留“同一容器、两种形态”的定义，用于对齐设计规则和实现能力。                     |
| `STATE_HALF_EXPANDED` 不是控件自身半高 | 新增    | 避免设计将“半展开”误解为视觉上展示一半内容，导致错误档位设计。                       |
| 标题栏可注册为拖动热区                    | 新增    | 与研发 `addDragHandleView` 能力对齐，解决 overlay 或固定标题栏场景的拖拽预期。 |
