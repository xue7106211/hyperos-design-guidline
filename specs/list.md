# 列表 List
## 一、概述
列表是一种按垂直顺序组织同类信息、入口或操作的组件，帮助用户快速浏览、比较、选择和处理内容。列表适用于设置项、消息、任务、联系人、设备和内容索引等场景。
- **识别性**：列表项应具备稳定的结构和清晰的主次关系，使用户能够快速识别内容、状态和可执行操作。
- **可扫描性**：列表应通过一致的对齐、间距、文本层级和分组方式降低阅读成本，避免用装饰替代有效信息。
- **层次性**：列表应根据内容重要性组织主文本、辅助文本、分组标题和右侧操作，保持同一列表中的信息结构一致。
- **可操作性**：可点击、可选择、可滑动和可编辑的列表项应提供明确的状态反馈，并保证操作入口可发现、可触达。
列表内容应优先以简洁、明确的文本呈现；图标、缩略图和装饰元素仅在能够帮助识别、理解或操作时使用。
## 二、类型与结构
### 2.1、展示类型
| 类型 | 用途 | 典型场景 | 关键约束 |
| --- | --- | --- | --- |
| 列表式 | 以连续行承载同类信息或入口 | 设置、消息、联系人、任务 | 列表项结构应保持一致，内容较多时可使用分组标题 |
| 卡片式 | 以独立卡片区分相互关联的信息 | 闹钟、设备、媒体、内容摘要 | 卡片之间应保持清晰间距，不应将不同结构的卡片强行放入同一组 |
| 分组列表 | 通过分组标题或分割关系组织多个内容集合 | 设置分类、消息日期、设备类别 | 分组标题应明确表达分组关系，不应仅作为装饰 |
| 横滑列表 | 在列表项横向滑动后展示辅助操作 | 删除、转发、收藏、标记 | 操作应与当前列表项直接相关，默认不超过 3 个主要操作 |
### 2.2、内容层级
| 结构 | 组成 | 适用场景 | 规则 |
| --- | --- | --- | --- |
| 单行列表 | 主文本 | 简短入口、单一状态或单一操作 | 主文本应独立表达完整含义 |
| 双行列表 | 主文本 + 辅助文本 | 消息摘要、设备说明、设置项解释 | 辅助文本用于补充信息，不应重复主文本 |
| 三行列表 | 主文本 + 辅助文本 + 状态或补充信息 | 内容摘要、复杂任务、消息详情 | 仅在三层信息都对任务有帮助时使用，避免为填充高度增加文本 |
### 2.3、结构组成
| 元素 | 必需 | 说明 |
| --- | --- | --- |
| 列表容器 | 是 | 承载列表项、分组标题和滚动内容；应与页面背景建立清晰层级。 |
| 列表项 | 是 | 承载一条完整信息、入口或操作；同一列表内应使用统一的结构和对齐方式。 |
| 主文本 | 是 | 表达列表项的核心对象或操作名称，应使用最高的信息强调层级。 |
| 辅助文本 | 可选 | 补充描述、时间、状态或来源；应位于主文本下方或与主文本保持明确关系。 |
| 左侧元素 | 可选 | 可使用徽标、系统图标、头像、预览图或应用图标帮助识别内容。 |
| 右侧元素 | 可选 | 可使用箭头、文本、图标、开关、复选框、下拉选择或功能按钮表达状态和操作。 |
| 分组标题 | 可选 | 用于划分不同内容集合；当列表内容存在明确分类时使用。 |
| 分割关系 | 可选 | 通过间距、分割线或背景区分列表项；不应同时使用过多分隔手段。 |
### 2.4、左右元素组合
左侧元素应服务于内容识别，右侧元素应服务于状态表达或操作触发。右侧可使用功能图标、文本，或图标与文本的组合；同一列表中相同语义的右侧元素应保持一致。
当右侧操作超过一个时，应按操作优先级排列，主要操作靠近内容，破坏性操作放在更易确认的位置，并通过颜色或图标表达语义。右侧元素与中间内容保持至少 `{spacing.list.trailingContent}` 的间距；具体数值待接入 HyperOS spacing token。
## 三、交互与状态
### 3.1、状态矩阵
| 状态 | 触发条件 | 视觉变化 | 行为规则 |
| --- | --- | --- | --- |
| 默认 | 列表项处于可用状态 | 使用默认背景、文字和图标层级 | 可按业务定义为可点击、可选择、可切换或仅展示 |
| 悬浮 | 指针位于可交互列表项上方 | 使用轻量悬浮背景或等效反馈 | 仅在支持指针输入的设备上展示，不应改变布局尺寸 |
| 按下 | 用户按住列表项或操作控件 | 使用即时按压反馈，保持内容位置稳定 | 释放后触发对应操作；取消按压不应执行操作 |
| 选中 | 用户完成单选或多选 | 使用选中背景、勾选或其他明确标识 | 选中状态应保持到用户取消选择、提交或离开当前选择任务 |
| 激活 | 当前项代表当前页面或当前上下文 | 使用激活色、指示条或其他持续标识 | 适用于分栏、导航和多窗口场景，不与瞬时按下态混用 |
| 聚焦 | 键盘、遥控器或辅助输入聚焦列表项 | 使用可辨识的焦点描边或焦点背景 | 焦点移动不应改变选中状态；焦点样式应满足对比度要求 |
| 禁用 | 当前列表项不可操作 | 降低文字、图标和控件强调度 | 不响应点击、滑动和选择，不应进入可操作焦点顺序 |
| 加载 | 数据尚未完成获取 | 使用骨架、进度指示或加载占位 | 加载占位应保持列表结构，避免内容完成后产生大幅跳动 |
| 空状态 | 当前列表无内容 | 展示简短说明和必要的引导操作 | 空状态应说明原因或下一步，不使用空白区域代替反馈 |
| 错误 | 数据获取或操作失败 | 展示错误说明和可重试入口 | 错误反馈应关联当前列表，不应覆盖用户已完成的操作结果 |
### 3.2、点击、选择与返回
- 可点击列表项的整行可作为主要点击区域；当行内包含独立控件时，应划分控件热区，避免点击控件同时触发整行操作。
- 点击列表项后，应根据任务进入详情、切换选中状态、展开内容或执行即时操作；同一列表中相同结构的列表项应保持一致的响应方式。
- 单选列表应明确当前选中项；多选列表应提供批量选择状态和退出选择模式的方式。
- 列表项需要进入下一级页面时，可在右侧使用箭头或等效的层级提示；仅展示状态时不应使用箭头制造可进入的暗示。
- 系统返回、页面切换或分栏切换后，应尽量保留列表的滚动位置、当前选中项和用户已完成的筛选状态。
### 3.3、横滑操作
横滑操作用于补充列表项的快捷行为，不应替代列表项的主要任务。操作区应使用语义明确的图标或简短文本，避免让用户通过颜色猜测操作含义。
| 规则 | 要求 |
| --- | --- |
| 操作数量 | 默认展示不超过 3 个主要操作；操作较多时应改用更多菜单、详情页或批量操作。 |
| 操作区宽度 | 展开后的操作区宽度不应超过列表项宽度的一半，避免挤压内容和误触。 |
| 触发距离 | 横滑触发距离通常取操作区宽度的一半；应支持回弹和关闭，不应停留在未定义的中间位置。 |
| 操作顺序 | 按频率和重要性排列；删除等不可逆操作应使用明确的破坏性样式，并在必要时二次确认。 |
| 滑动冲突 | 横滑与页面纵向滚动冲突时，应优先保证纵向滚动；可横滑区域应保持连续，不应被无关控件切断。 |
| 执行反馈 | 操作完成后应更新列表项、移除已删除内容或提供撤销入口，避免界面停留在过期状态。 |
### 3.4、滚动与长列表
- 长列表应支持连续滚动，并在数据分页、刷新或追加时保持已读状态、选中状态和滚动位置的连续性。
- 当列表项包含大量文本时，应优先拆分信息层级或拆分为多个列表项，不应依靠压缩字号和缩小间距承载过多内容。
- 列表项高度应根据内容复杂度确定；不同结构的列表项不应为了对齐而强制使用相同高度。
- 到达列表底部时，应明确区分“没有更多内容”和“仍在加载”；加载失败时提供局部重试，不应清空已有内容。
## 四、视觉规格
### 4.1、材质
| 属性 | Token / 规则 | 备注 |
| --- | --- | --- |
| 列表背景 | `{color.surface.list}` | 应与页面背景形成可辨识但不过度突出的层级 |
| 列表项背景 | `{color.surface.listItem}` | 默认状态不应依靠强阴影区分每一行 |
| 卡片背景 | `{color.surface.card}` | 卡片式列表使用独立表面和稳定的圆角层级 |
| 主文本 | `{color.text.primary}` | 用于核心内容、标题和主要入口 |
| 辅助文本 | `{color.text.secondary}` | 用于描述、时间、来源和次要状态 |
| 占位文本 | `{color.text.tertiary}` | 仅用于提示性或暂不可用内容 |
| 分割关系 | `{color.divider.list}` | 使用于需要明确区分内容的场景，避免与大间距重复表达 |
| 选中 / 激活 | `{color.state.selected}` | 应同时配合勾选、指示条或结构变化，不能只依赖颜色 |
| 禁用 | `{color.state.disabled}` | 通过文字、图标和控件的整体降级表达不可操作 |
| 破坏性操作 | `{color.action.destructive}` | 仅用于删除、移除等不可逆或高风险操作 |
### 4.2、文字与信息层级
- 列表应优先展示文本内容；主文本、辅助文本和状态信息应通过字号、字重、颜色和位置建立层级。
- 主文本应简洁完整，辅助文本应避免重复主文本；超长文本应按产品任务选择截断、换行或进入详情页。
- 分组标题可使用较小字号或较弱颜色，但必须与列表项正文形成稳定区分。
- 同一分组内的主文本、辅助文本和右侧状态应保持统一的对齐基线；不要将不同结构的列表项强行放入同一视觉分组。
### 4.3、组件规格
| 属性 | 规则 | 备注 |
| --- | --- | --- |
| 常用行高 | 48dp、56dp、64dp、72dp、96dp | 按内容复杂度选择，不应为填充空间增加无意义高度 |
| 右侧元素与内容间距 | `{spacing.list.trailingContent}` | 常用参考间距为 12dp，最终使用对应 token |
| 右侧文本与箭头间距 | `{spacing.list.textToChevron}` | 常用参考间距为 4dp，最终使用对应 token |
| 列表项内边距 | `{spacing.list.horizontalPadding}` | 应在同一列表中保持一致；特殊卡片可按容器层级调整 |
| 触控热区 | 不小于平台最小触控热区 | 图标视觉尺寸可小于热区，但热区不应缩小 |
| 卡片圆角 | `{radius.list.card}` | 与 HyperOS 卡片体系保持一致 |
| 分组间距 | `{spacing.list.group}` | 应大于列表项内部间距，明确表达分组关系 |
| 预览图 | 1:1、16:9、13:18、3:4 等 | 选择比例应由内容语义决定；不应在同一分组中混用无规则比例 |
### 4.4、左右元素档位
左侧元素应根据识别需要选择档位，右侧元素应根据操作语义选择控件，不应将所有可用元素同时放入一行。
| 元素 | 建议规格 |
| --- | --- |
| 徽标 | 8dp × 8dp |
| 系统图标 | 16dp 或 24dp |
| 头像 | 40dp、48dp 或 56dp |
| 1:1 预览图 | 48dp、56dp、72dp 或 96dp |
| 应用图标 | 48dp、56dp 或 64dp |
| 16:9 预览图 | 保持比例，最长边不超过 96dp |
| 13:18 / 3:4 预览图 | 小尺寸默认高度 64dp，大尺寸默认高度 96dp |
以上规格为常用展示档位；当内容需要更大视觉面积时，应在卡片式列表或详情场景中单独定义，不应直接放大普通列表行。
## 五、多端设计指南
| 终端 | 规则 |
| --- | --- |
| 手机 | 默认使用单列列表；小屏优先保留主文本和核心操作，可隐藏或折叠低优先级辅助信息。列表项的左右元素应适配安全区和单手触达范围。 |
| 折叠屏 | 根据折叠状态、窗口尺寸和任务关系在单列与双栏之间切换；展开后可将列表与详情并列，并保留当前选中项和滚动位置。 |
| 平板 | 优先使用响应式布局；内容较多时可采用列表 + 详情分栏，列表宽度不应被无限拉伸，主要信息区域应保持稳定的阅读宽度。 |
| 横屏窗口 | 可通过分栏或限制内容最大宽度适配宽窗口；列表项左右元素应保持原有对齐关系，不应因拉伸改变主次层级。 |
| 分屏窗口 | 所有列表尺寸、滚动范围和横滑操作应基于当前应用窗口计算，不应跨越分屏边界；窗口变窄时优先减少次要信息，而不是压缩核心操作热区。 |
| 指针与键盘设备 | 支持悬浮和聚焦状态；悬浮反馈不应改变布局，键盘操作应能按顺序移动焦点并完成选择、进入和返回。 |
