# 抽屉浮窗 Drawer Floating Window V2

## 一、概述

抽屉浮窗是一种从屏幕边缘进入、覆盖或并列于当前页面之上的容器型界面，用于承载临时任务、轻量页面、辅助配置或与背景内容并行的操作。

抽屉浮窗应被理解为同一容器在「抽屉形态」与「浮窗形态」之间的响应式切换，它在手机竖屏场景下通常表现为底部抽屉；在大屏、折叠屏、平板或横屏场景下，可根据断点与业务任务转化为浮窗。

## 二、核心原则

抽屉浮窗属于信息展示类基础组件中的「抽屉窗口」。以下原则为已确认的组件核心原则，应直接引用并作为后续规则判断依据。

- **非打扰性**：抽屉浮窗应在不打断主任务的前提下呈现补充信息、临时任务或并行配置。模态场景需要阻断背景时，应通过遮罩和返回规则清晰表达任务边界；非模态场景不应遮蔽或抢占背景操作。
- **层级与结构**：抽屉浮窗应明确表达自身与背景页面、内部标题区、内容区、操作区之间的层级关系。容器高度、遮罩、阴影、圆角、标题和把手应共同帮助用户理解当前信息结构。
- **上下文相关**：抽屉浮窗的内容应与发起入口或背景页面保持上下文关联。打开、关闭、内部跳转、系统返回和大屏形态转化都不应破坏用户对来源任务的理解。
- **连续性**：抽屉浮窗在展开、收起、档位切换、抽屉/浮窗形态切换、屏幕旋转和分屏变化中，应保持任务状态、内容位置和返回顺序连续，不应造成突兀跳变或状态丢失。
- **明确清晰**：抽屉浮窗应清晰传达当前任务、可操作区域、可关闭方式、拖拽热区和状态反馈。标题、操作文案、空/错/加载状态及手势反馈应避免歧义。



## 三、层级与类型


| 类型     | 背景关系   | 用途               | 典型场景                    | 关键约束                           |
| ------ | ------ | ---------------- | ----------------------- | ------------------------------ |
| 模态抽屉窗口 | 背景不可交互 | 临时展示少量信息或操作项     | 操作菜单、少量配置、确认前选择         | 应有遮罩；背景内容不可操作；关闭后返回发起页面。       |
| 模态抽屉页面 | 背景不可交互 | 以更轻量方式承载原本的临时全屏页 | 新建闹钟、编辑条目、跨进程打开轻量页、搜索选择 | 可展开到较大档位；系统返回应退出抽屉页面或回退内部页面。   |
| 非模态抽屉  | 背景可交互  | 与背景内容并行预览、编辑或配置  | 日历日程联动、小部件编辑、AOD 预览配置   | 不应有遮罩；背景可继续操作；抽屉高度变化可影响背景内容布局。 |
| 大屏浮窗   | 视任务而定  | 抽屉在大屏或横屏下的转化形态   | 平板、折叠屏、横屏、宽窗口           | 不应简单铺满底部；应按浮窗规则控制宽度、位置与层级。     |




### 3.1、形态策略


| 策略    | 规则                               | 设计含义                           |
| ----- | -------------------------------- | ------------------------------ |
| 自动响应式 | 默认策略，根据容器宽高、断点和业务类型在抽屉与浮窗之间自动切换。 | 推荐作为通用组件默认行为。                  |
| 固定抽屉  | 始终使用底部抽屉形态。                      | 仅适用于明确依赖底部进入、窄屏优先或业务尚未适配浮窗的场景。 |
| 固定浮窗  | 始终使用浮窗形态。                        | 适用于大屏优先、需要独立窗口感或不适合贴底展示的任务。    |


当形态从抽屉切换为浮窗时，任务语义、内部状态、返回顺序和内容结构应保持一致；可调整容器宽度、位置、把手展示和遮罩表现。浮窗形态默认不展示拖动把手，除非业务有明确的窗口拖动或高度调整需求。

### 3.2、类型选择规则


| 任务关系                | 推荐类型            | 判断依据                  |
| ------------------- | --------------- | --------------------- |
| 用户必须先处理浮层任务，背景仅作为来源 | 模态抽屉窗口 / 模态抽屉页面 | 背景不可交互，防止任务上下文被打断。    |
| 用户需要边看背景边配置当前任务     | 非模态抽屉           | 背景内容是预览或协作对象，不应被遮罩阻断。 |
| 内容是原全屏页的轻量调用        | 模态抽屉页面          | 保留页面结构，但以更轻量方式打开。     |
| 宽屏下底部抽屉阅读跨度过长       | 大屏浮窗            | 控制窗口宽度和视觉焦点。          |




## 四、结构与组成


| 元素   | 必需    | 说明                                        |
| ---- | ----- | ----------------------------------------- |
| 容器   | 是     | 承载标题区与内容区；应具备独立层级、圆角、背景、阴影或等效分层表现。        |
| 遮罩   | 视类型而定 | 模态抽屉应提供遮罩并阻断背景交互；非模态抽屉不应提供遮罩。             |
| 滑动把手 | 可选    | 用于提示可拖拽、切换高度或关闭；当仅有一个固定档位且不可关闭时，可不展示。     |
| 标题区  | 可选    | 可包含标题、关闭、取消、确认或其他页面级操作；规则应继承标题栏，但不支持大字标题。 |
| 内容区  | 是     | 展示核心内容、表单、列表、宫格、选择器或配置项；底部按钮栏属于内容区的一部分。   |
| 操作区  | 可选    | 当存在主操作、确认、取消等底部操作时，应与内容滚动区域保持清晰边界。        |




### 4.1、把手展示模式


| 模式         | 规则                       | 适用场景                                |
| ---------- | ------------------------ | ----------------------------------- |
| 占位模式       | 把手占据容器顶部一行空间，内容区从把手下方开始。 | 默认场景，内容顶部不需要贴边展示。                   |
| Overlay 模式 | 把手叠加在内容之上，不额外挤占内容区高度。    | 内容区顶部有图片、封面、ActionBar 或需要贴近面板边缘的视觉。 |


开启 Overlay 模式时，把手样式应使用专门适配 overlay 的视觉参数；如果业务自定义把手样式，应继承基础 overlay 样式，避免丢失暗色、热区、对比度等默认适配。

## 五、交互与状态

### 5.1、状态矩阵


| 状态   | 触发条件             | 视觉变化                     | 行为规则                |
| ---- | ---------------- | ------------------------ | ------------------- |
| 默认   | 抽屉已打开且可交互        | 容器、标题区、内容区按默认层级展示        | 用户可操作内容或切换档位。       |
| 拖拽中  | 用户拖动把手、标题区或内容区顶部 | 容器跟随手势移动；背景遮罩或并行区域可按进度变化 | 释放后吸附到目标档位或关闭。      |
| 最大档位 | 抽屉已到 L 档或业务最大高度  | 顶部仍保留抽屉形态与安全间距           | 继续上滑仅回弹，不应进入无定义高度。  |
| 最小档位 | 抽屉已到收起态或业务最小高度   | 展示最小可识别内容或入口             | 继续下滑可关闭，除非业务定义不可关闭。 |
| 加载中  | 内容尚未完成加载         | 内容区展示加载反馈，框架保持稳定         | 不应因加载完成造成突兀档位跳变。    |
| 空状态  | 内容为空或暂无结果        | 内容区展示空状态说明与可选行动          | 不应关闭抽屉代替空状态说明。      |
| 错误状态 | 加载失败或权限不足        | 内容区展示错误信息与重试/引导          | 不应让用户停留在无反馈空白容器。    |
| 禁用关闭 | 任务必须完成或不可中断      | 关闭入口隐藏或禁用，并提供原因或替代操作     | 应谨慎使用，避免造成困住用户。     |




### 5.2、研发状态映射


| 设计语义         | BottomSheet 状态        | 说明                                            |
| ------------ | --------------------- | --------------------------------------------- |
| 收起态          | `STATE_COLLAPSED`     | 面板保留最小可见高度或入口；是否允许跳过由业务配置。                    |
| M 档 / 半展开参考态 | `STATE_HALF_EXPANDED` | 注意该状态不是“显示控件自身高度的一半”，而是控件顶部距离窗口顶部约半个窗口高度。     |
| L 档 / 展开态    | `STATE_EXPANDED`      | 面板进入展开状态；当内容或配置达到 full height 时，表现为贴近顶部的统一状态。 |
| 关闭 / 隐藏      | `STATE_HIDDEN`        | 仅当业务允许隐藏或关闭时可进入。                              |


`STATE_HALF_EXPANDED` 只应在内容区自身高度达到 full height 阈值，或业务强制 full height 时启用；否则交互上应跳过半展开态，避免面板脱离底部或停在不符合预期的位置。

### 5.3、打开与关闭


| 行为   | 模态抽屉                            | 非模态抽屉                                    |
| ---- | ------------------------------- | ---------------------------------------- |
| 打开   | 从发起控件或底部边缘进入，背景进入遮罩与不可交互状态。     | 从底部边缘进入，背景保持可见且可交互。                      |
| 关闭按钮 | 可提供关闭、取消或返回入口。                  | 若抽屉中途被拉出或可被移除，应提供关闭入口；若抽屉始终存在且不可关闭，可不提供。 |
| 点击遮罩 | 默认存在遮罩，点击遮罩等同关闭操作。              | 不存在遮罩。                                   |
| 系统返回 | 应关闭抽屉并回到发起页面。                   | 应按业务定义关闭抽屉或返回背景页面，需避免不可预期。               |
| 手势关闭 | 在最小档位继续下滑可关闭；高风险任务应要求显式确认或保存提示。 | 可关闭场景遵循同样规则；不可关闭场景不应响应关闭手势。              |




### 5.4、档位切换


| 操作区域        | 上滑规则                                          | 下滑规则                                        |
| ----------- | --------------------------------------------- | ------------------------------------------- |
| 标题区 / 把手    | 未到最大档位时，短上滑切换到临近更大档位，长上滑切换到最大档位；已到最大档位时仅轻微回弹。 | 未到最小档位时，短下滑切换到临近更小档位，长下滑可关闭；已到最小档位时继续下滑可关闭。 |
| 内容区顶部或无滚动容器 | 优先扩展抽屉档位；无更大档位时再滚动内容或触发回弹。                    | 优先收缩抽屉档位；无更小档位时关闭抽屉。                        |
| 内容区中间       | 优先滚动内容，不应抢占为抽屉高度变化。                           | 优先滚动内容回到顶部；到顶部后再触发抽屉收缩或关闭。                  |




### 5.5、拖动热区


| 热区              | 规则                                   |
| --------------- | ------------------------------------ |
| 把手              | 默认可作为整体拖动入口；浮窗形态默认不展示把手。             |
| 标题区 / ActionBar | 建议注册为拖动热区，尤其是 overlay 模式或固定标题栏场景。    |
| 内容区顶部 / 底部      | 仅当内容滚动到边界时，才可将纵向滑动转交给抽屉整体拖动。         |
| 普通内容区           | 不建议默认触发整体拖动，避免与列表、表单、横滑组件、地图等内容交互冲突。 |


拖动热区中的点击与拖拽应按位移阈值区分：未超过阈值时保留原视图点击、长按等行为；超过阈值后由抽屉接管并跟随手指移动。

## 六、使用指南



### 6.1、应做（Do）

- 应在需要保留背景上下文的临时任务中使用抽屉浮窗，而不是直接跳转全屏页。
- 应根据背景是否可继续交互，明确选择模态或非模态类型。
- 应为每个业务入口定义默认档位、最大档位、最小档位和是否可关闭。
- 应在横屏、折叠屏和平板场景评估使用底部抽屉或转化为浮窗，横屏允许继续使用底部抽屉。
- 应在内容滚动和抽屉拖拽之间建立明确手势优先级。
- 应在未保存内容、错误、空状态和权限不足时给出清晰反馈。



### 6.2、不应做（Don't）

- 不应在同一页面同时打开多个抽屉浮窗，除非已明确定义层级和返回顺序。
- 不应把复杂全屏流程强行塞入抽屉，导致导航、返回和状态保存不清晰。
- 不应让非模态抽屉带遮罩，或让模态抽屉背景仍可操作。
- 不应把 `STATE_HALF_EXPANDED` 理解为“展示内容的一半”。
- 不应让普通内容区任意位置都触发整体拖动，避免与内容交互冲突。
- 不应为了 edge-to-edge 视觉关闭系统避让后，遗漏底部导航栏、小白条、IME 和安全区适配。



## 七、变体详解



### 7.1、模态抽屉窗口

用于临时展示少量信息或操作项，通常不需要复杂内部导航。

#### 常见使用场景

- 操作菜单、轻量配置、确认前选择。
- 系统能力入口，如分享、图片选择、文件选择等封装能力。



#### 布局

- 默认使用底部抽屉形态。
- 内容较少时可自适应高度；内容超过 M 档容量时应进入 L 档或使用抽屉页面。
- 应提供遮罩并阻断背景交互。



#### 约束

- 关闭后应返回发起页面。
- 默认存在遮罩，点击遮罩等同关闭操作。



### 7.2、模态抽屉页面

用于以更轻量方式打开原本的临时全屏页，保留页面级标题区和内部导航能力。

#### 常见使用场景

- 新建、编辑、跨进程调用轻量页、搜索选择。



#### 布局

- 可展开到 L 档。
- 承载全屏页时，标题栏建议使用导航吸顶样式。
- 内部页面跳转时，抽屉框架、遮罩关系和把手规则应保持稳定。



#### 约束

- 系统返回应优先回退抽屉内部层级；无内部层级时关闭抽屉。
- 不建议在抽屉内再嵌套新的模态抽屉。



### 7.3、非模态抽屉

用于并行任务，背景可继续预览或操作，抽屉与背景形成协同关系。

#### 常见使用场景

- 日历和日程协同、小部件编辑、AOD 预览配置、背景预览同时编辑。



#### 布局

- 不应展示遮罩。
- 抽屉高度变化可影响背景内容布局；背景被遮挡区域应重排或明确不可见。
- 单档且不可关闭时可隐藏把手。



#### 约束

- 若抽屉中途被拉出或可被移除，应提供关闭入口。
- 背景与抽屉焦点顺序应可预测。



### 7.4、大屏浮窗

用于抽屉在横屏、折叠屏、平板或宽窗口下的转化形态。

#### 常见使用场景

- 手机横屏、折叠屏展开态、平板、应用分屏宽窗口。



#### 布局

- 宽度和位置应按浮窗规则控制，不应横向铺满。
- 若业务强制保持抽屉形态，建议在空间足够时水平居中，并使用与浮窗一致或接近的宽度。



#### 约束

- 形态切换不应改变任务语义、内部状态和返回顺序。
- 浮窗形态默认不展示拖动把手。



## 八、规格



### 8.1、色彩


| 属性   | Token / 规则                   | 备注                           |
| ---- | ---------------------------- | ---------------------------- |
| 容器背景 | 应与当前页面背景形成清晰层级，并支持浅色/深色模式。   | 待接入 `{color.surface.*}`。     |
| 遮罩   | 模态抽屉使用遮罩降低背景强调层级；非模态抽屉不使用遮罩。 | 待接入 `{color.overlay.*}`。     |
| 标题文字 | 使用标题栏对应文字层级，不支持大字标题。         | 待接入 `{font.title.*}`。        |
| 把手   | 低强调、可识别，不应与标题或主操作竞争。         | 待接入 `{color.icon.tertiary}`。 |




### 8.2、布局


| 场景        | 规则                                           |
| --------- | -------------------------------------------- |
| 手机竖屏      | 默认从底部弹出，宽度贴合当前窗口宽度。                          |
| 手机横屏      | 允许继续使用底部抽屉；也可根据业务需要转化为浮窗。                    |
| 手机横屏但保持抽屉 | 空间足够时水平居中，并使用与浮窗一致或接近的宽度。                    |
| 折叠屏 / 平板  | 模态抽屉窗口与模态抽屉页面建议转化为浮窗；非模态抽屉可继续以半屏/分栏形态服务并行任务。 |
| 左右分屏      | 每个应用分屏区域可独立使用抽屉浮窗；不应跨越当前应用窗口边界。              |
| 上下分屏      | 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。                 |




### 8.3、尺寸


| 档位    | 规则                                                  | 适用                  |
| ----- | --------------------------------------------------- | ------------------- |
| 收起态   | 半屏高度 = 半屏内标题区的高度。                                   | 非模态抽屉、少数可持续存在的工具面板。 |
| M 档   | 半屏高度 = 屏幕高度 * 70%。                                  | 多数默认半屏任务。           |
| L 档   | 半屏高度 = 屏幕高度 - 状态栏 - 间距。                             | 抽屉页面、复杂内容、长列表、表单编辑。 |
| 自适应高度 | 内容在 S-M 之间时，根据内容多少自适应高度，最低不低于 XX%；内容超过 M 时直接使用 L 档。 | 少量内容的模态抽屉窗口。        |
| 自定义档位 | 由业务定义默认高度、最大高度、最小高度与切换逻辑。                           | 非模态抽屉、需要背景预览的编辑工具。  |


高度变化应遵循有限档位，不支持无极悬停。若内容因大字号或多语言超过 M 档容量，建议直接进入 L 档。

### 8.4、间距

- L 档顶部应保留状态栏、安全区与顶部间距。
- 键盘弹出时，抽屉应避让键盘，保证当前输入项与主操作可见。
- 底部操作区应与内容滚动区域保持清晰边界。
- 具体间距待接入 HyperOS spacing token。



### 8.5、圆角


| 形态   | 规则                        |
| ---- | ------------------------- |
| 底部抽屉 | 顶部边缘应体现浮层容器形态；底部贴合窗口或安全区。 |
| 浮窗   | 应使用四角圆角，并与浮窗体系保持一致。       |


圆角数值待接入 `{radius.*}`。

### 8.6、阴影 / 海拔

抽屉浮窗应表达高于页面内容的 Z 轴层级；非模态抽屉尤其需要清晰区分抽屉与背景内容。具体海拔待接入 `{elevation.*}`。

### 8.7、热区

- 把手、标题区或 ActionBar 可作为拖动热区。
- 关闭、取消、完成等操作应满足平台最小触控热区。
- 图标通常不随文本等比例放大，但触控热区不应缩小。



## 九、内容与文案

- 标题应说明当前任务，而不是重复发起按钮文案。
- 关闭、取消、完成、保存等操作文案应与任务结果一致；存在未保存内容时应明确提示。
- 空状态、错误状态和权限状态应放在内容区内处理，不应改变抽屉类型。
- 多语言长标题应遵从标题栏适配规则；业务不应依赖固定字数保证布局。



## 十、动效规范

- 打开与关闭应使用上下方向的进入/退出动效，来源方向与抽屉位置一致。
- 档位切换应吸附到明确高度，并给出自然回弹；不应停留在未定义的中间高度。
- 模态抽屉打开时，遮罩透明度应与容器进入同步变化。
- 非模态抽屉调整高度时，背景内容若需要避让或重排，应与抽屉运动协调完成。
- 本规范不涉及具体动画 token、时长、曲线与降级策略。



## 十一、响应式变化规则



### 11.1、高度规则

- 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。
- 内容超过 M 档容量时，直接使用 L 档，不应无限增高。
- 内容区自身高度未达到 full height 阈值时，应跳过 `STATE_HALF_EXPANDED`。
- full height 状态下若希望内容区撑满剩余空间，应显式配置内容高度匹配父容器。



### 11.2、宽度规则

- 手机竖屏默认贴合当前窗口宽度。
- 横屏允许继续使用底部抽屉；具体宽度策略可按业务和后续断点规则确定。
- 折叠屏和平板断点与窗口最大宽度待单独补充；横屏允许继续使用底部抽屉。
- 分屏中不应跨越当前应用窗口边界。



### 11.3、位置规则

- 底部抽屉从底部进入并吸附在底部安全区之上。
- 浮窗形态应按浮窗主题规则控制位置与大小。
- Edge-to-edge 场景如关闭系统自动避让，业务必须自行处理底部导航栏、小白条、IME 与安全区。



### 11.4、多项排布

- 底部操作如果存在多个按钮，应根据可用宽度和多语言长度决定横排或纵排。
- 同一界面在所有多语言下，操作区布局建议保持一致。



## 十二、多端设计指南


| 终端       | 规则                                  |
| -------- | ----------------------------------- |
| 手机       | 竖屏优先使用底部抽屉；横屏允许继续使用底部抽屉，也可按业务转化为浮窗。 |
| 折叠屏      | 折叠窄屏可沿用手机规则；展开态应评估浮窗、分栏或非模态半屏。      |
| 平板       | 模态任务优先使用浮窗；并行任务可保留非模态半屏或分栏式抽屉。      |
| 分屏       | 每个分屏窗口独立计算宽高、位置和安全区；不可跨窗口展示。        |
| 键鼠 / 触控板 | 需支持焦点、键盘返回、点击 outside 和滚轮/触控板滚动仲裁。  |




## 十三、无障碍与可用性

- 模态抽屉打开后，焦点应限制在抽屉内部；关闭后焦点应回到发起控件。
- 非模态抽屉不应阻断背景焦点，但焦点顺序需在背景与抽屉之间可预测。
- 关闭、取消、完成等操作必须具备可读标签，不应只依赖图标。
- 把手若支持拖拽，应提供等效的键盘、开关控制或明确的可访问替代操作。
- 文本最大放大倍数参考旧资料为 1.45x；最终应与 HyperOS 字号无障碍策略确认。
- 图标通常不随文本等比例放大，需保证触控热区仍满足平台最小要求。



## 十四、边界场景


| 场景           | 处理规则                                           |
| ------------ | ---------------------------------------------- |
| 内容过长         | 内容区应滚动；当超过 M 档容量时直接使用 L 档，不应无限增高。              |
| 内容过少         | 可使用自适应高度；内容在 S-M 之间时根据内容多少自适应，高度最低不低于 XX%。     |
| 未保存更改        | 关闭前应提示保存、放弃或继续编辑。                              |
| 键盘弹出         | 抽屉应避让键盘，保证当前输入项与主操作可见。                         |
| Edge-to-edge | 如业务追求沉浸式贴边效果，应自行处理底部导航栏、小白条、IME 与安全区，不应牺牲可操作性。 |
| 内部滚动嵌套       | 应定义主滚动容器，避免多层纵向滚动导致手势冲突。                       |
| 背景可交互但被遮挡    | 非模态抽屉应让背景内容根据抽屉高度调整布局或明确被覆盖区域不可见。              |
| 多窗口叠加        | 不建议同时打开多个抽屉；若有系统弹窗叠加，应明确最高层级与返回顺序。             |




## 十五、实现说明


| 项目   | 说明                                                                                                                             |
| ---- | ------------------------------------------------------------------------------------------------------------------------------ |
| 研发承接 | Android / MIUIX 场景优先以 `BottomSheetModal` 与 `BottomSheetBehavior` 承接；`BottomSheetDialog` / `BottomSheetDialogFragment` 不作为推荐实现。 |
| 类型声明 | 组件应显式声明 `modal` / `nonModal` / `floating`，不应仅由是否展示遮罩反推类型。                                                                      |
| 形态策略 | 默认使用自动响应式策略；固定抽屉或固定浮窗需有业务理由。                                                                                                   |
| 档位配置 | 业务应提供默认高度、最大高度、最小高度、是否可关闭、是否可拖拽等配置。                                                                                            |
| 手势仲裁 | 实现需区分标题区拖拽、内容区滚动、内容区顶部下拉和系统返回手势。                                                                                               |


| 状态保存   | 档位变化、未保存输入、加载状态不应因旋转、分屏变化或窗口尺寸变化丢失。                                                                                            |
| 推荐实现   | Android / MIUIX 应优先使用 `BottomSheetModal`；其以非模态实现承接模态效果，可避免 `Dialog` / `DialogFragment` 的窗口动画、内存开销和高级材质限制。                      |
| 展示前状态  | 每次展示前应明确设置初始状态，避免复用上一次状态导致高度或形态异常。                                                                                             |
| 主题继承   | 业务自定义背景、把手或样式时应继承 MIUIX BottomSheet 基础 style，不应完全重写导致默认配置丢失。                                                                   |
| Insets | 默认应主动避让底部系统装饰与 IME；若关闭自动避让以实现 edge-to-edge，业务必须自行处理 WindowInsets。                                                              |
| 内容撑满   | full height 状态下若希望内容区撑满剩余空间，应显式配置内容高度匹配父容器。                                                                                    |

## 十六、QA 检查清单

- [ ] 模态抽屉打开后，背景不可点击、不可聚焦，系统返回可关闭或回退内部页面。
- [ ] 非模态抽屉打开后，背景仍可按预期预览或操作，且无遮罩。
- [ ] 标题区、内容区顶部、内容区中间的上滑/下滑优先级符合规范。
- [ ] 标题区或 ActionBar 已作为拖动热区接入，且点击与拖拽不会互相误触。
- [ ] M/L/收起态/自定义档位能正确吸附，不停留在未定义高度。
- [ ] `STATE_HALF_EXPANDED` 只在 full height 条件满足时出现，不会造成面板脱离底部。
- [ ] 长内容、大字号、多语言、空状态、错误状态不会破坏布局。
- [ ] 键盘弹出时，输入项和主操作仍可见且可操作。
- [ ] Edge-to-edge 场景已处理底部导航栏、小白条、IME 与安全区。
- [ ] 手机竖屏、横屏、折叠屏、平板、分屏下的形态转化符合业务定义。
- [ ] 深色模式与浅色模式下，容器层级、遮罩和文字可读性一致。



## 十七、从 OS3 迁移


| 旧规范规则                          | V2 处理 | 原因                                                     |
| ------------------------------ | ----- | ------------------------------------------------------ |
| 模态抽屉用于临时任务，背景有遮罩且不可交互          | 保留    | 仍是区分临时任务与并行任务的核心规则。                                    |
| 非模态抽屉用于并行任务，背景可交互              | 保留并强化 | HyperOS 多任务、大屏与预览编辑场景更依赖此规则。                           |
| 抽屉高度遵循低、中、高三个尺寸，不支持无极悬停        | 调整    | V2 改写为收起态、M、L、自适应与业务自定义档位，但仍强调有限档位吸附。                  |
| 横屏暂不支持，强制竖屏                    | 调整    | V2 明确横屏允许继续使用底部抽屉，也可按业务转化为浮窗。                          |
| 折叠屏、PAD 默认为浮窗，小部件特殊适配          | 保留并泛化 | V2 将其扩展为大屏转化规则，允许非模态抽屉按并行任务保留半屏形态。                     |
| 标题栏规则与全屏页面一致，但不支持大字标题          | 保留    | 维持页面级导航一致性，同时避免抽屉标题区占用过高。                              |
| 非模态单档且不可关闭时可隐藏把手               | 保留    | 避免把手暗示不存在的拖拽或关闭能力。                                     |
| 点击遮罩不可关闭窗口                     | 调整    | V2 明确模态抽屉默认存在遮罩，点击遮罩等同关闭操作。                            |
| 研发 BottomSheet 支持抽屉与浮窗形态切换     | 新增    | V2 将抽屉浮窗定义为同一容器的响应式双形态，便于设计规则与实现能力对齐。                  |
| `STATE_HALF_EXPANDED` 不是控件自身半高 | 新增    | 避免设计将“半展开”误解为视觉上展示一半内容，导致错误档位设计。                       |
| 标题栏可注册为拖动热区                    | 新增    | 与研发 `addDragHandleView` 能力对齐，解决 overlay 或固定标题栏场景的拖拽预期。 |
