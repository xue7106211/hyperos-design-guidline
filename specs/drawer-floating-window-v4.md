# 抽屉浮窗 Drawer Floating Window V4

## 一、概述

抽屉浮窗是从屏幕边缘进入的容器，用来承载临时任务、轻量任务、辅助配置，也可以和背景内容并行工作。

同一个抽屉浮窗可以在「抽屉形态」和「浮窗形态」之间切换。手机竖屏通常使用底部抽屉；大屏、折叠屏、平板和横屏下，可按断点和任务需要转为大抽屉或浮窗。

## 二、核心原则

- **非打扰性**：抽屉浮窗用于呈现补充信息、临时任务或并行配置，尽量不打断主任务。模态场景用遮罩和返回规则说明任务边界；非模态场景不要遮蔽或抢占背景操作。
- **层级与结构**：抽屉浮窗需要让用户看清它和背景页面、标题区、内容区、操作区之间的层级。容器高度、遮罩、阴影、圆角、标题和把手共同表达这层结构。
- **上下文相关**：抽屉浮窗的内容要和发起入口或背景页面有关联。打开、关闭、内部层级切换、系统返回和大屏转化，都要保留这条来源线索。
- **连续性**：展开、收起、档位切换、抽屉/浮窗切换、屏幕旋转和分屏变化时，任务状态、内容位置和返回顺序要保持连续，避免突然跳变或状态丢失。
- **明确清晰**：抽屉浮窗需要说清当前任务、可操作区域、关闭方式、拖拽热区和状态反馈。标题、操作文案、空状态、错误状态、加载状态和手势反馈都要避免歧义。

## 三、类型

抽屉浮窗只分两类：**模态抽屉**和**非模态抽屉**。先看背景能不能继续操作，再决定要不要转成大抽屉或浮窗。内容多不算一种新类型，只会影响同一类型下的高度档位和展示形态。
这里的“大抽屉”和“浮窗”都只是形态，不是新的类型。浮窗只适用于大屏设备，而且只出现在模态抽屉里。

### 3.1、类型判断

| 类型 | 背景关系 | 典型任务 | 默认形态 | 大屏形态 | 语义 |
| --- | --- | --- | --- | --- | --- |
| 模态抽屉 | 背景不可交互 | 临时确认、一次性选择、需要先完成再回到原任务的操作 | 底部抽屉 | 大抽屉或浮窗 | 任务要先结束或确认，再回到原上下文 |
| 非模态抽屉 | 背景可交互 | 边看背景边预览、编辑或配置 | 底部抽屉或半屏 | 大抽屉或分栏形态 | 任务和背景并行进行，互不阻断 |

## 四、结构与组成

| 元素   | 必需    | 说明                                        |
| ---- | ----- | ----------------------------------------- |
| 容器   | 是     | 承载标题区与内容区；应具备独立层级、圆角、背景、阴影或等效分层表现。        |
| 遮罩   | 视类型而定 | 模态抽屉应提供遮罩并阻断背景交互；非模态抽屉不应提供遮罩。             |
| 滑动把手 | 可选    | 用于提示可拖拽、切换高度或关闭；当仅有一个固定档位且不可关闭时，可不展示。     |
| 标题区  | 可选    | 可包含标题、关闭、取消、确认或其他页面级操作；规则应继承标题栏，但不支持大字标题。 |
| 内容区  | 是     | 展示核心内容、表单、列表、宫格、选择器或配置项；底部按钮栏也算内容区。       |
| 操作区  | 可选    | 有主操作、确认、取消等底部操作时，要和内容滚动区域分清。              |


### 4.1、把手展示模式

| 模式         | 规则                       | 适用场景                                |
| ---------- | ------------------------ | ----------------------------------- |
| 占位模式       | 把手占据容器顶部一行空间，内容区从把手下方开始。 | 默认场景，内容顶部不需要贴边展示。                   |
| Overlay 模式 | 把手叠加在内容之上，不额外挤占内容区高度。    | 内容区顶部有图片、封面、ActionBar 或需要贴近面板边缘的视觉。 |

开启 Overlay 模式后，把手使用 overlay 适配样式。业务自定义把手时，需要继承基础 overlay 样式，避免丢失暗色、热区、对比度等默认适配。

### 4.2、浮窗部分

浮窗部分当前只承接模态抽屉的大屏形态，不单列独立类型。它的结构组成以容器、遮罩、标题区和内容区为主。

| 元素 | 必需 | 说明 |
| --- | --- | --- |
| 容器 | 是 | 承载浮窗整体；应具备独立层级、背景、圆角和阴影或等效分层表现。 |
| 遮罩 | 是 | 浮窗只用于模态场景，应提供遮罩并阻断背景交互。 |
| 标题区 | 可选 | 承载标题、关闭入口，必要时可放返回或确认等页面级操作；首次使用介绍等场景可不展示标题栏。 |
| 内容区 | 是 | 展示核心内容、表单、列表、配置项或预览内容。 |

## 五、交互与状态

这一章只写抽屉容器的交互：5.1 定义高度档位和可达关系，5.2 定义拖拽过程反馈，5.3 定义退出与返回语义，5.4 定义叉号与勾号的操作语义。加载、空状态、错误状态属于内容区反馈，不作为抽屉状态单独定义。

### 5.1、高度档位与停靠状态

| 设计档位 | 高度定义 |
| --- | --- |
| S 档 / 折叠态 | 业务定义的最小可见高度或入口高度。 |
| M 档 / 半展开态 | 可见高度为容器高度的 70%，不是内容高度或控件自身高度的一半。 |
| L 档 / 展开态 | 面板展开到业务最大高度或接近全高；full height 场景下为统一贴顶状态。 |
| 自定义高度 | 内容高度介于 S 档与 M 档之间时，根据内容多少自适应，最低不低于 XX%。 |
| 关闭 / 隐藏 | 面板离开可视区域。 |

本节只定义高度档位和可达关系。业务应明确默认档位、最大档位、最小档位，以及是否允许关闭。S、M、L 是抽屉的主要高度档位；M 档只在内容达到 full height 阈值或业务强制 full height 时可用，不满足条件时应跳过，避免控件脱离底部。业务允许关闭时，关闭 / 隐藏可以作为任一可关闭场景的目标状态，不要求必须经过 M 档或 S 档。自定义高度不是独立停靠状态，只是展开态下的内容自适应高度；内容超过 M 档容量时直接使用 L 档，不在 M-L 之间创建额外状态。

内容区加载完成、结果为空或发生错误时，抽屉应保持当前合法档位。只有业务明确要求重新定位时，才调整抽屉高度。

### 5.2、拖拽、吸附与回弹

| 阶段 | 触发条件 | 容器表现 | 行为规则 |
| --- | --- | --- | --- |
| 待交互 | 抽屉停在稳定位置 | 容器、标题区、内容区按默认层级展示 | 用户可操作内容，容器等待拖拽输入。 |
| 拖拽中 | 抽屉拖拽被触发 | 容器跟随手势移动；遮罩或背景区域可随进度变化 | 位移未超过阈值时保留原点击、长按；超过阈值后由抽屉接管。 |
| 吸附中 | 用户释放拖拽 | 容器过渡到目标位置 | 根据位移、速度和方向判断目标；不停在中间高度。 |
| 边界回弹 | 用户继续向可移动范围外拖动 | 容器出现轻微阻尼或回弹 | 回弹只提示已到边界，不产生新的目标位置。 |

拖拽反馈要连续。手指移动、容器位置、遮罩透明度和背景关系应同步变化。释放后的目标位置必须来自业务定义的可达范围。

### 5.3、关闭、返回与中断保护

| 退出方式 | 模态抽屉 | 非模态抽屉 | 规则 |
| --- | --- | --- | --- |
| 关闭按钮 | 可提供关闭、取消或返回入口 | 可关闭场景应提供明确入口；常驻抽屉可不提供 | 入口文案应说明操作后果。 |
| 点击遮罩 | 默认等同关闭 | 不适用，非模态抽屉不使用遮罩 | 高风险任务关闭前应确认或提示保存。 |
| 系统返回 | 关闭抽屉，或先回退抽屉内部层级 | 按业务定义关闭抽屉或返回背景上下文 | 返回顺序应可预期，不跳过未保存提示。 |
| 任务完成 | 提交结果后关闭抽屉并回到发起上下文 | 保存配置或结束并行任务后关闭抽屉 | 完成、取消、关闭的结果应区分清楚。 |

任务必须完成、存在未保存内容或操作不可中断时，可以限制关闭。限制关闭时要给出原因或替代操作，避免用户被困在抽屉里。

### 5.4、叉号与勾号定义

抽屉浮窗顶部的叉号和勾号应先定义语义，再决定位置。同一个叉号可能表示取消，也可能表示关闭；判断核心是：点击叉号是在取消一件正在进行的事，还是只是结束一次信息查看。

| 控件 | 语义 | 位置 | 适用场景 | 规则 |
| --- | --- | --- | --- | --- |
| 叉号 / 取消 | Cancel，放弃当前任务或退出正在编辑的流程 | 左侧 | 用户已经进入选择、填写、编辑、创建等任务流程，退出会放弃本次操作或未提交结果 | 左侧叉号应被理解为“取消”；如存在未保存内容或高风险放弃结果，应给出确认或保存提示。 |
| 返回 | Back，回到抽屉内上一层级或上一步 | 左侧 | 抽屉内部存在二级层级、编辑子页、选择子页或可回退流程 | 返回只回退层级，不应等同于关闭整个抽屉；没有上一级时不展示返回。 |
| 退出 | Exit，离开当前流程或中止当前模式 | 左侧 | 录音、控制中心、备忘录等存在“退出当前编辑 / 录制 / 模式”的场景 | 退出语义比取消更偏向离开当前模式；若退出会丢失结果，应给出确认或保存提示。 |
| 叉号 / 关闭 | Close，结束信息查看或收起展示层 | 右侧 | 抽屉只承载信息展示、详情查看、说明预览等，不涉及编辑、提交或任务结果 | 右侧叉号应被理解为“关闭”；关闭后不应丢失业务结果，也不应让用户误以为取消了某个提交。 |
| 勾号 / 确认 | Confirm，确认当前选择或当前步骤 | 右侧 | 用户需要确认一项选择、设置或一次轻量结果 | 确认用于认可当前结果，不等同于保存或完成；结果重要时优先使用文字按钮。 |
| 勾号 / 完成 | Done，完成当前编辑或添加流程 | 右侧 | 编辑、添加、创建流程已经完成，需要结束当前任务 | 完成表示任务已结束，并回到发起上下文或上一层级。 |
| 勾号 / 保存 | Save，保存当前修改 | 右侧 | 设置、天气、世界时钟等需要保留用户改动的场景 | 保存应只在有可保存结果时出现；保存后按业务返回上一层级或关闭抽屉。 |
| 勾号 / 应用 | Apply，应用当前设置但不一定关闭任务 | 右侧 | 锁屏、桌面等设置预览或配置场景 | 应用表示让当前配置生效；是否关闭抽屉由业务流程决定。 |

当抽屉同时存在取消和确认时，取消放左侧，确认放右侧。不要在同一个顶部区域同时放置两个含义相近的关闭入口，也不要让右侧叉号承担取消编辑、放弃创建等任务语义。

当顶部只有一个叉号时，应按任务语义判断位置：如果点击叉号会放弃正在进行的流程，叉号放左侧；如果点击叉号不会放弃任何结果，只是关闭信息展示层，叉号可放右侧。

所有顶部操作都应配置清晰的无障碍语义。TalkBack 读出的功能名称应与实际行为一致，例如“取消”“返回”“退出”“关闭”“确认”“完成”“保存”或“应用”，不能只读出图标形状。

## 六、使用指南

### 6.1、应做（Do）

- 临时任务需要保留背景上下文时，使用抽屉浮窗，不直接跳全屏页。
- 先判断背景是否还能操作，再选择模态或非模态。
- 每个业务入口都要定义默认档位、最大档位、最小档位，以及是否可关闭。
- 横屏、折叠屏和平板要判断继续用底部抽屉，还是转为大抽屉或浮窗；横屏可以继续使用底部抽屉。
- 内容滚动和抽屉拖拽之间要有明确的手势优先级。
- 未保存内容、错误、空状态和权限不足，都要给出反馈。

### 6.2、不应做（Don't）

- 同一业务入口不要同时打开多个抽屉浮窗，除非已经定义好层级和返回顺序。
- 不要把复杂全屏流程塞进抽屉。导航、返回和状态保存会变得不清楚。
- 非模态抽屉不要带遮罩；模态抽屉打开后，背景不要继续操作。
- 不要把 `STATE_HALF_EXPANDED` 理解为“展示内容的一半”。
- 普通内容区不要任意位置都触发整体拖动，避免和内容交互冲突。
- 如果为了 edge-to-edge 关闭系统避让，需要同时处理底部导航栏、小白条、IME 和安全区。

## 七、规格

### 7.1、色彩

| 属性   | Token / 规则                   | 备注                           |
| ---- | ---------------------------- | ---------------------------- |
| 容器背景 | 应与当前界面背景形成清晰层级，并支持浅色/深色模式。   | 待接入 `{color.surface.*}`。     |
| 遮罩   | 模态抽屉使用遮罩降低背景强调度；非模态抽屉不使用遮罩。 | 待接入 `{color.overlay.*}`。     |
| 标题文字 | 使用标题栏对应文字层级，不支持大字标题。         | 待接入 `{font.title.*}`。        |
| 把手   | 低强调、可识别，不应与标题或主操作竞争。         | 待接入 `{color.icon.tertiary}`。 |

### 7.2、布局

| 场景        | 规则                                           |
| --------- | -------------------------------------------- |
| 手机竖屏      | 默认从底部弹出，宽度贴合当前窗口宽度。                          |
| 手机横屏      | 允许继续使用底部抽屉；也可根据业务需要转化为浮窗。                    |
| 手机横屏但保持抽屉 | 空间足够时水平居中，并使用与浮窗一致或接近的宽度。                    |
| 折叠屏 / 平板  | 模态抽屉建议转化为大抽屉或浮窗；非模态抽屉可继续以半屏/分栏形态服务并行任务。 |
| 左右分屏      | 每个应用分屏区域可独立使用抽屉浮窗；不跨越当前应用窗口边界。               |
| 上下分屏      | 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。                 |

### 7.3、尺寸

| 档位    | 规则                                                  | 适用                  |
| ----- | --------------------------------------------------- | ------------------- |
| S 档 / 折叠态 | 半屏高度 = 半屏内标题区的高度。                                   | 非模态抽屉、少数可持续存在的工具面板。 |
| M 档   | 半屏高度 = 屏幕高度 * 70%。                                  | 多数默认半屏任务。           |
| L 档   | 半屏高度 = 屏幕高度 - 状态栏 - 间距。                             | 内容较多的模态抽屉、复杂内容、长列表、表单编辑。 |
| 自定义高度 | 内容在 S-M 之间时，根据内容多少自适应高度，最低不低于 XX%；内容超过 M 时直接使用 L 档。 | 少量内容的模态抽屉。        |
| 自定义档位 | 由业务定义默认高度、最大高度、最小高度与切换逻辑。                           | 非模态抽屉、需要背景预览的编辑工具。  |

高度变化应遵循有限档位，不支持无极悬停。若内容因大字号或多语言超过 M 档容量，建议直接进入 L 档。

### 7.4、间距

- L 档顶部应保留状态栏、安全区与顶部间距。
- 键盘弹出时，抽屉应避让键盘，保证当前输入项与主操作可见。
- 底部操作区应与内容滚动区域保持清晰边界。
- 具体间距待接入 HyperOS spacing token。

### 7.5、圆角

| 形态   | 规则                         |
| ---- | -------------------------- |
| 底部抽屉 | 顶部边缘需要有浮层容器的形态；底部贴合窗口或安全区。 |
| 浮窗   | 应使用四角圆角，并与浮窗体系保持一致。        |

圆角数值待接入 `{radius.*}`。

### 7.6、热区

- 把手、标题区或 ActionBar 可作为拖动热区。
- 关闭、取消、完成等操作应满足平台最小触控热区。
- 图标通常不随文本等比例放大，但触控热区不应缩小。

## 八、动效规范

- 第一层抽屉浮窗打开时，应自下而上弹出；关闭时沿相反方向退出。
- 第二级层级默认不触发新的抽屉浮窗动效；无特殊要求时，应作为抽屉浮窗内二级层级从右向左划入，并在左上角提供返回图标。
- 档位切换应吸附到明确高度，并给出自然回弹；不要停在未定义的中间高度。
- 模态抽屉打开时，遮罩透明度应与容器进入同步变化。
- 非模态抽屉调整高度时，背景内容若需要避让或重排，应与抽屉运动协调完成。
- 本规范不定义具体动画 token、时长、曲线和降级策略。

### 8.1、多重抽屉浮窗覆盖

当当前抽屉浮窗需要触发新的抽屉浮窗覆盖时，新抽屉浮窗应作为新的临时任务覆盖在旧抽屉浮窗之上，而不是作为旧抽屉浮窗内部层级推进。

适合使用多重抽屉浮窗覆盖的场景：

- 新内容是独立临时任务，有自己的标题、关闭入口和完成结果。
- 新任务不是旧任务的下一步，而是从旧任务中临时打开的选择、确认、授权、系统能力或跨流程任务。
- 关闭新任务后，用户应回到旧抽屉浮窗原状态继续操作。
- 新任务需要独立处理系统返回、未保存提示或关闭确认。
- 当前抽屉浮窗不是 L 规格，但新任务需要完整覆盖旧任务以保证可读性和操作空间。

不适合使用多重抽屉浮窗覆盖的场景：

- 新层级只是同一任务流程的下一步、详情、编辑或结果。
- 新层级需要继承当前任务状态、返回栈和完成逻辑。
- 关闭操作应直接退出整个临时任务，而不是只关闭当前新层级。
- 层级关系可以通过标题栏返回图标清楚表达。

| 项目 | 规则 |
| --- | --- |
| 标题栏 | 新抽屉浮窗显示自身临时任务标题栏，不继承上一个抽屉浮窗标题。 |
| 动效 | 新抽屉浮窗从底部弹出并覆盖旧抽屉浮窗；旧抽屉浮窗可降低强调度，并建议做轻微缩小后退。 |
| 系统返回 | 先响应新抽屉浮窗，直到新抽屉浮窗关闭后，再响应旧抽屉浮窗。 |
| 关闭抽屉浮窗 | 按层级退出，先关闭新抽屉浮窗，再关闭旧抽屉浮窗。 |
| 高度规格 | 当在非 L 规格的抽屉浮窗中触发新的抽屉浮窗覆盖时，新抽屉浮窗统一使用 L 规格，以完整覆盖旧抽屉浮窗。 |

### 8.2、抽屉浮窗内层级切换

当任务仍属于同一个抽屉浮窗内的层级变化时，应保持抽屉浮窗容器不动，只切换容器内层级。

适合使用抽屉浮窗内层级切换的场景：

- 新层级是同一临时任务的下一步、详情、编辑、搜索结果或确认结果。
- 层级之间共享同一标题栏体系、任务状态、表单数据或完成逻辑。
- 用户点击返回时应回到上一级层级，而不是关闭一个独立抽屉浮窗。
- 用户点击关闭时应退出整个抽屉浮窗任务。

| 项目 | 规则 |
| --- | --- |
| 标题栏 | 每个层级都应提供返回入口，通常为左上角返回图标。 |
| 动效 | 抽屉浮窗容器保持不动；层级从右向左前进，从左向右后退。 |
| 系统返回 | 逐级返回上一层级，直到没有内部层级后，再关闭抽屉浮窗。 |
| 关闭抽屉浮窗 | 不论当前处于哪个内部步骤，关闭操作都应直接关闭整个抽屉浮窗，退出当前临时任务。 |

## 九、多端设计指南

### 9.1、高度规则
- 抽屉高度应基于当前应用窗口高度计算，而不是物理屏幕高度。
- 内容超过 M 档容量时，直接使用 L 档，不要无限增高。
- 内容区自身高度未达到 full height 阈值时，应跳过 `STATE_HALF_EXPANDED`。
- full height 状态下若希望内容区撑满剩余空间，应显式配置内容高度匹配父容器。

### 9.2、宽度规则
- 手机竖屏默认贴合当前窗口宽度。
- 横屏允许继续使用底部抽屉；具体宽度策略可按业务和后续断点规则确定。
- 折叠屏和平板断点与窗口最大宽度待单独补充；横屏允许继续使用底部抽屉。
- 分屏中不跨越当前应用窗口边界。

### 9.3、位置规则
- 底部抽屉从底部进入并吸附在底部安全区之上。
- 浮窗形态应按浮窗主题规则控制位置与大小。
- Edge-to-edge 场景如关闭系统自动避让，业务必须自行处理底部导航栏、小白条、IME 与安全区。

### 9.4、多项排布
- 底部操作如果存在多个按钮，应根据可用宽度和多语言长度决定横排或纵排。
- 同一界面在所有多语言下，操作区布局建议保持一致。

### 9.5、多端规则
| 终端       | 规则                                   |
| -------- | ------------------------------------ |
| 手机       | 竖屏优先使用底部抽屉；横屏允许继续使用底部抽屉，也可按业务转化为浮窗。  |
| 折叠屏      | 折叠窄屏可沿用手机规则；展开态应评估大抽屉、浮窗、分栏或非模态半屏。       |
| 平板       | 模态任务优先使用大抽屉或浮窗；并行任务可保留非模态半屏或分栏式抽屉。       |
| 分屏       | 每个分屏窗口独立计算宽高、位置和安全区；不跨窗口展示。          |
| 键鼠 / 触控板 | 需要支持焦点、键盘返回、点击 outside，以及滚轮/触控板滚动仲裁。 |

## 十、QA 检查清单

- [ ] 模态抽屉打开后，背景不可点击、不可聚焦，系统返回可关闭或回退内部层级。
- [ ] 非模态抽屉打开后，背景仍可按预期预览或操作，且无遮罩。
- [ ] 标题区、内容区顶部、内容区中间的上滑/下滑优先级符合规范。
- [ ] 标题区或 ActionBar 已作为拖动热区接入，且点击与拖拽不会互相误触。
- [ ] S/M/L/关闭状态能正确切换，自定义高度按展开态的内容自适应高度处理。
- [ ] `STATE_HALF_EXPANDED` 只在 full height 条件满足时出现，不会造成面板脱离底部。
- [ ] 长内容、大字号、多语言、空状态、错误状态不会破坏布局。
- [ ] 键盘弹出时，输入项和主操作仍可见且可操作。
- [ ] Edge-to-edge 场景已处理底部导航栏、小白条、IME 与安全区。
- [ ] 新抽屉浮窗覆盖旧抽屉浮窗时，新抽屉浮窗使用 L 规格，系统返回和关闭都先作用于新抽屉浮窗。
- [ ] 抽屉浮窗内二级层级切换时，容器保持不动，层级左右进退，关闭入口可直接退出整个抽屉浮窗任务。
- [ ] 手机竖屏、横屏、折叠屏、平板、分屏下的形态转化符合业务定义。
- [ ] 深色模式与浅色模式下，容器层级、遮罩和文字可读性一致。

## 十一、从 OS3 迁移

| 旧规范规则                          | V4 处理 | 原因                                                     |
| ------------------------------ | ----- | ------------------------------------------------------ |
| 模态抽屉用于临时任务，背景有遮罩且不可交互          | 保留    | 仍是区分临时任务与并行任务的核心规则。                                    |
| 非模态抽屉用于并行任务，背景可交互              | 保留并强化 | HyperOS 多任务、大屏与预览编辑场景更依赖此规则。                           |
| 抽屉高度遵循低、中、高三个尺寸，不支持无极悬停        | 调整    | V4 使用 S/M/L/关闭作为稳定状态；自定义高度是展开态下的内容自适应高度，不作为独立停靠状态。                    |
| 横屏暂不支持，强制竖屏                    | 调整    | V4 明确横屏可以继续使用底部抽屉，也可以按业务转为大抽屉或浮窗。                          |
| 折叠屏、PAD 默认为浮窗，小部件特殊适配          | 保留并泛化 | V4 将它整理为大屏转化规则；非模态抽屉可按并行任务保留半屏形态。                      |
| 标题栏规则与全屏界面一致，但不支持大字标题          | 保留    | 维持界面级导航一致性，同时避免抽屉标题区占用过高。                              |
| 非模态单档且不可关闭时可隐藏把手               | 保留    | 避免把手暗示不存在的拖拽或关闭能力。                                     |
| 点击遮罩不可关闭抽屉                     | 调整    | V4 明确模态抽屉默认有遮罩，点击遮罩就是关闭。                               |
| 研发 BottomSheet 支持抽屉与浮窗形态切换     | 新增    | V4 保留“同一容器、两种形态”的定义，用于对齐设计规则和实现能力；多端可承接大抽屉或浮窗。                     |
| `STATE_HALF_EXPANDED` 不是控件自身半高 | 新增    | 避免设计将“半展开”误解为视觉上展示一半内容，导致错误档位设计。                       |
| 标题栏可注册为拖动热区                    | 新增    | 与研发 `addDragHandleView` 能力对齐，解决 overlay 或固定标题栏场景的拖拽预期。 |
