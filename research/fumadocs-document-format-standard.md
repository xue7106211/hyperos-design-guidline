# 设计规范文档内容与排版标准

## 一、目的

本标准用于统一设计规范在 Fumadocs 网站中的内容结构与呈现方式，确保设计师、开发、产品、测试以及 AI 都能快速完成以下任务：

- 理解设计标准及其适用场景；
- 查询明确的尺寸、间距、状态和行为规格；
- 根据规则判断设计方案是否符合规范；
- 追溯判断依据、例外条件和关联规则；
- 在不同终端宽度下保持良好的阅读体验。

本标准主要约束 Markdown 表格、图片、Tabs、Accordion、Cards、Callout 等内容形式的使用方式。

## 二、基本原则

### 2.1、一种形式只解决一种问题

- 表格用于结构化比较，不用于承载长篇解释。
- 图片用于展示视觉效果，不作为规格的唯一来源。
- Tabs 用于切换互斥视图，不用于隐藏核心规则。
- Accordion 用于补充信息，不用于收纳必须阅读的内容。
- Callout 用于强调风险、例外和状态，不用于代替正文结构。

### 2.2、优先保证独立可读

每个章节、表格和规则单元应尽可能独立成立。读者通过搜索或锚点进入页面中部时，不应依赖前文才能理解当前内容。

为保证独立可读，可以适度重复类型、状态、平台或对象名称，不应通过合并单元格或省略上下文来减少文字。

### 2.3、关键信息必须以文本表达

以下内容不得只存在于图片、Tabs、Accordion 或视觉布局关系中：

- 强制规则；
- 具体规格；
- 适用条件；
- 判断依据；
- 关键例外；
- 验收标准；
- 版本状态。

## 三、表格使用标准

### 3.1、适用场景

Markdown 表格适合表达：

- 类型之间的差异；
- 状态、触发条件与行为的对应关系；
- 属性与规格值；
- 平台之间的规则差异；
- 版本迁移与变更记录；
- 简短、结构一致的验收条件。

以下内容不适合使用表格：

- 需要连续阅读的设计原理；
- 包含多段解释的复杂规则；
- 包含多个步骤的操作流程；
- 需要大量图片辅助的案例；
- 条件、原因、例外和验证方式都较复杂的规则。

### 3.2、列数限制

| 列数 | 使用要求 |
| --- | --- |
| 2～4 列 | 推荐，可直接使用标准 Markdown 表格。 |
| 5 列 | 谨慎使用，仅限每列内容简短且维度不可继续拆分的情况。 |
| 6 列及以上 | 不应直接使用，必须拆分维度或更换呈现形式。 |

即使表格没有超过 4 列，如果单元格普遍包含多句话、多个判断条件或多段内容，也应进行拆分。

### 3.3、行数限制

- 一般表格建议不超过 12 个数据行。
- 超过 12 行时，应优先按照类型、状态、平台或场景拆分。
- Token 清单、版本记录等天然属于长列表的内容可以例外，但应使用小标题分组。
- 不应通过 Accordion 隐藏过长表格来回避信息拆分。

### 3.4、单元格内容限制

- 一个单元格只表达一个信息维度。
- 单元格内容建议不超过两句话。
- 不应在单元格中放置图片、列表、Callout 或代码块。
- 不应在单元格中书写多段正文。
- 不应使用空单元格表达无内容，应统一填写 `—`、`不适用` 或 `待确认`。
- 同一列的单位、语气和信息结构应保持一致。
- 表头必须准确描述判断维度，避免使用“其他”“补充”“说明 2”等含糊名称。

### 3.5、表格结构要求

- 第一列应明确比较对象，例如类型、状态、平台或属性。
- 其余各列应分别对应一个稳定维度。
- 同一张表不应同时比较类型、平台、状态和角色等多个层级。
- 数值单位应统一写在表头或各规格值中，不应混用。
- 规则正文中引用表格内容时，应使用明确的表格标题或规则 ID。

## 四、多维表格拆分方法

### 4.1、按信息性质拆分

当表格同时包含视觉、行为、无障碍和实现信息时，应拆成独立表格。

不推荐：

```markdown
| 状态 | 触发条件 | 视觉变化 | 行为规则 | 无障碍 | 实现限制 |
| --- | --- | --- | --- | --- | --- |
```

推荐：

```markdown
### 状态表现

| 状态 | 触发条件 | 视觉变化 |
| --- | --- | --- |

### 状态行为

| 状态 | 行为规则 | 无障碍要求 | 实现限制 |
| --- | --- | --- | --- |
```

两张表可以重复“状态”列。信息重复优先于关系不清。

### 4.2、改为纵向属性表

描述单一对象的大量规格时，应将横向矩阵改为两列或三列属性表。

```markdown
| 属性 | 规格 | 适用条件 |
| --- | --- | --- |
| 高度 | 48 dp | 默认尺寸 |
| 左右间距 | 16 dp | 手机端标准窗口 |
| 圆角 | 12 dp | 默认容器 |
| 点击热区 | 不小于 48 × 48 dp | 所有可交互入口 |
```

纵向属性表适合表达：

- 单个组件的尺寸；
- 单个状态的详细规格；
- Token 映射；
- 动效参数；
- 单个平台的适配规则。

### 4.3、按平台或角色拆分

手机、折叠屏、平板等互斥视图可以使用 Tabs 分组，每个 Tab 内继续使用不超过 4 列的表格。

```mdx
<Tabs items={['手机', '折叠屏', '平板']}>
  <Tab value="手机">
    手机端规格。
  </Tab>

  <Tab value="折叠屏">
    折叠屏规格。
  </Tab>

  <Tab value="平板">
    平板端规格。
  </Tab>
</Tabs>
```

使用 Tabs 时应遵循：

- 默认规则应写在 Tabs 外部。
- Tab 内只呈现平台差异或角色补充。
- 强制规则不得只存在于某个 Tab 中。
- Tab 数量建议不超过 4 个。
- 需要连续阅读的步骤或流程不应使用 Tabs 拆分。

### 4.4、改为规则单元

当一条规则需要同时说明条件、要求、原因、验证方式和例外时，应使用规则单元，不应继续扩展表格列数。

```markdown
### R-BS-012 背景交互

- **适用条件**：抽屉用于模态任务。
- **要求**：抽屉显示期间，背景内容不得操作。
- **判断依据**：用户需要先完成或取消当前任务。
- **验证方式**：背景控件无法点击、聚焦或被读屏访问。
- **例外**：无。
```

规则单元应具有稳定标题或 Rule ID，便于链接、搜索、引用和 AI 解析。

## 五、合并单元格标准

### 5.1、禁止使用合并单元格

标准规范正文不应使用合并单元格，包括通过 HTML 实现的 `rowspan` 和 `colspan`。

合并单元格依赖视觉位置表达信息关系，容易造成：

- 窄屏下阅读关系不明确；
- 复制到其他工具后结构丢失；
- AI 难以识别单元格归属；
- 无障碍阅读顺序复杂；
- 后续增删内容时结构容易被破坏。

### 5.2、替代方法

需要表达分组关系时，应依次考虑：

1. 重复填写分类名称；
2. 使用二级或三级标题分组；
3. 拆成多个独立表格；
4. 使用两列属性表；
5. 将公共规则写在表格前；
6. 将复杂内容改写为规则单元。

不推荐通过合并单元格组织多个状态：

```text
模态抽屉 ─ 默认
         ├ 展开
         └ 关闭
```

推荐使用标题和独立表格：

```markdown
### 模态抽屉

| 状态 | 行为规则 | 视觉表现 |
| --- | --- | --- |
| 默认 | … | … |
| 展开 | … | … |
| 关闭 | … | … |
```

## 六、图片使用标准

### 6.1、适用场景

图片适合展示：

- 组件结构；
- 视觉效果；
- 布局关系；
- 手势与动效过程；
- 正确与错误案例；
- 难以用文字准确描述的空间关系。

图片不应代替：

- 尺寸和间距规格；
- 状态名称；
- 使用条件；
- 行为规则；
- 正确或错误的判断原因；
- 无障碍与验收标准。

### 6.2、图片数量

- 一个三级章节原则上使用 0～2 张图片。
- Do/Don’t 对比优先使用一组两张图片。
- 连续出现 3 张以上图片时，应拆分为多个场景或使用图集。
- 同一规则不应重复使用多张高度相似的截图。
- 表格内部不应插入图片。
- 长页面中不应连续堆叠多个没有说明的全宽大图。

图片数量不是唯一判断依据。如果一张图片同时表达过多状态、规格和说明，也应拆分。

### 6.3、图片说明

每张图片应包含：

- 准确的替代文本 `alt`；
- 图片标题或图注；
- 图片所表达的目的；
- 对应的规则 ID 或正文说明；
- 正确或错误案例的判断原因。

```markdown
![非模态抽屉允许用户继续操作背景内容](/images/bottom-sheet/non-modal-preview.png)

*图 3：非模态抽屉用于边查看背景、边调整配置的任务。对应规则 `R-BS-021`。*
```

图片中的关键文字、数值和结论必须在正文中重复表达，不能要求读者或 AI 只通过识图获取规则。

## 七、MDX 组件使用标准

| 组件 | 适合承载 | 不适合承载 |
| --- | --- | --- |
| Markdown 表格 | 2～4 个维度的精确比较 | 长篇解释、复杂层级关系 |
| Tabs | 平台、角色、互斥变体 | 核心规则、默认规格 |
| Accordion | 补充说明、历史信息、低频例外 | 强制要求、关键验收条件 |
| Cards | 页面入口、相关规范导航 | 精确规格、判断依据 |
| Callout | 风险、废弃、待确认、重要例外 | 普通正文、完整规则体系 |
| 图片 | 视觉效果、结构示意、正反案例 | 唯一的尺寸或行为定义 |
| 列表 | 原则、步骤、检查项 | 多对象、多维度比较 |

### 7.1、Tabs

Tabs 只用于读者一次查看其中一个分组的场景，例如平台、角色或互斥变体。

以下内容不得只放在 Tabs 中：

- 核心原则；
- 默认规则；
- 强制规格；
- 关键例外；
- 验收标准。

### 7.2、Accordion

Accordion 适合收纳：

- 历史版本说明；
- 低频边界场景；
- 补充解释；
- 迁移细节；
- 不影响主要判断的实现备注。

Accordion 不应用于减少正文长度而隐藏关键内容。

### 7.3、Callout

Callout 应只用于需要明显提示的信息：

- `重要`：忽略后可能产生明显体验问题；
- `警告`：存在高风险或不可逆结果；
- `待确认`：信息尚未完成校准；
- `废弃`：规则或方案已停止使用；
- `例外`：允许偏离默认规则的特殊条件。

同一章节不应连续出现多个同级 Callout。普通规则应写入正文或规则单元。

## 八、面向 AI 的内容要求

为了让 AI 能够识别和判断规范，文档应遵循：

- 标题层级稳定，不通过字号或图片模拟标题。
- 每个章节只讨论一个明确主题。
- 每条关键规则使用稳定 Rule ID。
- 使用“必须”“不得”“应”“建议”“可”等统一规则级别。
- 明确写出适用条件、规则要求、判断依据、验证方式和例外。
- 不依赖表格位置、颜色、合并单元格或图片关系传递语义。
- 图片内的重要信息必须同步为正文文本。
- 同一概念应使用统一名称，避免频繁使用简称或同义词。
- 不确定的信息统一标记为“待确认”，不得与正式规则混写。

AI 可以读取完整的结构化规范，不需要单独设置“AI”Tab。

## 九、推荐的规范页面组合

一篇完整的组件或模式规范可以按照以下顺序组合内容形式：

1. **概述**：一至两段正文；
2. **核心原则**：三至五条列表；
3. **类型选择**：决策树或不超过 4 列的表格；
4. **结构与组成**：三列表格配一张结构图；
5. **交互与状态**：四列状态矩阵；
6. **关键规则**：带 Rule ID 的规则单元；
7. **规格**：按照色彩、尺寸、间距、圆角和热区拆分的属性表；
8. **多端差异**：公共规则加平台 Tabs；
9. **使用案例**：Do/Don’t 图片与文字判断；
10. **QA**：可勾选检查清单；
11. **待确认与变更记录**：独立章节。

## 十、发布前检查清单

### 表格

- [ ] 所有表格均不超过 4 列，或已经说明保留 5 列的必要性。
- [ ] 没有 6 列及以上的标准 Markdown 表格。
- [ ] 没有合并单元格或 HTML `rowspan`、`colspan`。
- [ ] 单元格中没有图片、列表、Callout、代码块或多段正文。
- [ ] 空值已统一写为 `—`、`不适用` 或 `待确认`。
- [ ] 每张表只比较一个层级的问题。

### 图片

- [ ] 图片不是规格或判断标准的唯一载体。
- [ ] 每张图片都有准确的替代文本和图注。
- [ ] 图片中的关键数值和结论已同步到正文。
- [ ] 没有在表格内部插入图片。
- [ ] 没有连续堆叠多张缺少说明的图片。

### MDX 组件

- [ ] 强制规则没有只存在于 Tabs 或 Accordion 中。
- [ ] Tabs 只用于平台、角色或互斥变体。
- [ ] Accordion 只用于补充和低频信息。
- [ ] Callout 只用于重要、警告、例外、废弃或待确认信息。

### 可理解与可判断

- [ ] 读者可以快速找到具体规格。
- [ ] 规则明确说明适用条件和判断依据。
- [ ] 关键规则具有稳定标题或 Rule ID。
- [ ] 页面在窄屏下不依赖横向滚动才能理解主要内容。
- [ ] AI 不通过识图或视觉位置关系也能理解关键规则。

## 十一、核心约束摘要

1. 表格默认不得超过 4 列。
2. 5 列表格必须经过内容评审，6 列及以上必须拆分。
3. 标准规范正文禁止使用合并单元格。
4. 单元格不放置图片、列表、代码块或多段正文。
5. 同一张表只解决一个比较问题。
6. 图片不得成为规格和判断标准的唯一载体。
7. 强制规则不得只存在于 Tabs 或 Accordion 中。
8. 每条关键规则必须能够被单独引用、搜索和识别。
