# 面向设计师、开发、产品与 AI 的多维设计规范文档模式

> 研究日期：2026-07-17  
> 适用场景：基于 Fumadocs 构建的 HyperOS 设计规范网站

## 一、结论

推荐采用 **规则驱动的分层文档（Rule-centered Layered Documentation）**：

1. 只维护一份权威规则，避免按角色复制四套内容。
2. 页面先提供快速判断，再逐步展开规格、原因和验证方法。
3. 每条关键规则使用稳定 ID 和固定字段，既方便人查阅，也方便 AI 检索与引用。
4. 设计师、开发、产品使用角色视图进入同一份规则；AI 读取结构化 Markdown，而不是单独维护“AI 版文档”。
5. Fumadocs 负责页面导航、Tabs、搜索、MDX 渲染和 LLM 文本输出；自定义组件负责规则卡、决策矩阵和验收清单。

这不是一种新的视觉皮肤，而是一种 **信息模型 + 页面层级 + 多入口视图** 的组合。

---

## 二、现有线性规范的问题

当前模板已经覆盖概述、原则、类型、结构、状态、使用指南、规格、多端、QA 和更改日志，内容维度较完整。主要问题在于：

- 所有信息按照章节顺序线性展开，读者必须先知道答案“可能在哪一章”。
- “规则是什么”“为什么这样规定”“如何验收”混在同一叙述层级。
- 设计师、开发和产品对同一规则关心的字段不同，但当前没有角色入口。
- 表格描述了信息，却未必直接支持“当前场景应该选什么”的判断。
- AI 能读取全文，但缺少稳定规则 ID、明确条件、例外和验证标准，回答时容易只截取局部文字。
- 图片、Do/Don't 示例和规格表之间缺少可追踪关系。

因此，下一步不应继续增加章节，而应重新组织“规则的最小单位”和“阅读顺序”。

---

## 三、四类主体真正需要什么

| 主体 | 首要问题 | 最有效的内容形式 |
| --- | --- | --- |
| 设计师 | 什么时候用、选哪种、视觉和状态如何设计 | 决策树、类型矩阵、结构图、状态图、Do/Don't、规格表 |
| 开发 | 必须实现什么、状态如何变化、边界和多端规则是什么 | 行为规则、状态转换、Token、约束、异常场景、验收标准 |
| 产品 | 为什么使用、解决什么问题、适用范围和风险是什么 | 场景判断、业务前提、模式选择、关键限制、例外说明 |
| AI | 如何准确找到规则、判断适用条件并给出可追溯答案 | 原子规则、稳定 ID、显式字段、版本状态、结构化 Markdown |

四类主体并不是需要四份不同事实，而是需要对同一事实采用不同的读取顺序。

## 四、三种文档模式对比

### 方案 A：按角色拆分文档

为设计师、开发、产品和 AI 分别建立页面。

**优点**

- 每类读者看到的内容很直接。
- 页面可以使用角色熟悉的语言。

**问题**

- 同一规则会被复制，多份内容容易不一致。
- 修改规范时需要同步多处。
- 角色边界并不稳定，设计师也需要行为信息，开发也需要理解设计意图。
- “AI 版正文”会形成新的维护负担。

**判断**：不适合作为主结构，只适合做聚合入口或筛选视图。

### 方案 B：按内容类型拆分文档

按照概念、指南、规格、验收等内容类型组织站点，接近 Diátaxis 的 Explanation、How-to、Reference 等思路。

**优点**

- 不同类型内容的表达目的清楚。
- 规格参考可以保持客观、统一、易检索。
- 适合大型文档站的信息架构。

**问题**

- 一个组件的完整信息可能散落在多个页面。
- 读者完成一次具体判断时，需要跨页面往返。
- 设计规范除了“理解”和“查询”，还有高频的“选择与判定”任务。

**判断**：适合站点级导航，不足以单独解决组件页内部结构。

### 方案 C：规则驱动的分层文档

每个组件或模式仍保留一个主页面，但页面内部围绕同一组权威规则提供不同层次和入口。

**优点**

- 一份规则真源，多种阅读方式。
- 同时支持快速决策、深入理解、精确查规格和验收。
- 稳定规则 ID 便于搜索、链接、评审、变更追踪和 AI 引用。
- 与现有 Markdown/MDX 内容兼容，可渐进迁移。

**问题**

- 需要定义规则数据结构和少量自定义 MDX 组件。
- 写作者需要从“写章节”转向“定义规则单元”。

**判断**：推荐作为 HyperOS 规范站的核心模式。

## 五、推荐页面结构

一篇组件规范建议分成六层。读者不需要按顺序读完，而是按任务进入。

### 第 1 层：页面身份与可信度

位于标题附近，用于回答“这是什么、是否有效、适用于哪里”。

建议展示：

- 规范状态：草稿 / 评审中 / 已发布 / 已废弃
- 当前版本与最后复核日期
- 适用平台：手机、折叠屏、平板、桌面等
- 负责人或维护团队
- 关联组件、Token、研发文档
- 变更提示

### 第 2 层：30 秒判断摘要

页面开头不再直接进入长篇概述，而是提供一个 `DecisionSummary`。

应包含：

- **用它解决什么问题**
- **何时使用**
- **何时不要使用**
- **最关键的类型选择**
- **三个不可违反的规则**

这一层主要服务产品和首次进入页面的设计师、开发。

### 第 3 层：选择与判断

使用决策树、场景矩阵或步骤回答“我当前应该选什么”。

推荐形式：

```text
是否阻断背景操作？
├─ 是 → 模态
│  ├─ 内容是否超过 M 档？→ L 档
│  └─ 内容较少 → 自适应高度或 M 档
└─ 否 → 非模态
   ├─ 是否需要持续预览背景？→ 保留半屏/分栏形态
   └─ 是否转入独立任务？→ 考虑全屏页或浮窗
```

决策树只负责选型，不承载所有规格细节；每个结论应链接到对应规则 ID。

### 第 4 层：权威规则

这是整篇文档的核心事实层。每条关键规则使用统一结构表达。

建议的规则单元：

| 字段 | 说明 |
| --- | --- |
| Rule ID | 稳定标识，例如 `drawer.modal.background-interaction` |
| 规则级别 | 必须 / 不得 / 应 / 建议 / 可 |
| 适用条件 | 规则在什么场景下生效 |
| 规则正文 | 明确、可执行、无歧义的要求 |
| 判断依据 | 用户目标、设计原则、平台能力或风险 |
| 对设计的影响 | 类型、视觉、布局、状态或内容要求 |
| 对实现的影响 | 行为、状态、响应式、无障碍或限制 |
| 验证方式 | 可观察、可测试的验收条件 |
| 例外 | 哪些情况可以偏离，以及由谁确认 |
| 关联项 | 相关规则、Token、示例、变更记录 |

规则示例：

```mdx
<Rule
  id="drawer.modal.background-interaction"
  level="不得"
  audiences={["design", "development", "product", "ai"]}
  status="stable"
>
  <Rule.When>抽屉用于需要用户先完成或取消的临时任务时。</Rule.When>
  <Rule.Requirement>模态抽屉显示期间，背景内容不得继续操作。</Rule.Requirement>
  <Rule.Rationale>避免用户同时改变两个上下文，导致任务状态和返回关系不明确。</Rule.Rationale>
  <Rule.Design>使用遮罩，并提供明确的完成、取消或关闭路径。</Rule.Design>
  <Rule.Engineering>背景区域不接收指针、键盘和无障碍焦点。</Rule.Engineering>
  <Rule.Verify>打开抽屉后，无法点击、聚焦或通过读屏访问背景控件。</Rule.Verify>
</Rule>
```

### 第 5 层：角色视图

角色视图不是独立正文，而是对权威规则的聚合和补充。

可使用 Fumadocs Tabs：

- **设计**：类型选择、结构、视觉规格、状态、Do/Don't
- **开发**：行为、状态转换、Token、响应式、无障碍、限制
- **产品**：适用场景、业务前提、模式选择、风险、例外
- **验收**：可执行检查项、失败表现、覆盖场景

不建议设置“AI”Tab。AI 应读取完整的结构化规则，而不是读取被人为摘要后的角色视图。

Tabs 适合放角色补充信息，但以下内容不应只存在于 Tab 内：

- 核心规则
- 强制规格
- 关键例外
- 版本状态
- 验收标准

### 第 6 层：参考与追溯

页面后部保留传统参考内容：

- 结构与组成
- 状态矩阵
- 规格与 Token
- 动效
- 多端规则
- 无障碍
- 完整 QA 清单
- 迁移说明
- 更改日志

这部分适合精确查询，应使用稳定标题、表格和锚点，避免长篇混合叙述。

## 六、页面视觉骨架

```text
┌──────────────────────────────────────────────┐
│ 标题 / 状态 / 版本 / 平台 / 最后复核          │
├──────────────────────────────────────────────┤
│ 30 秒判断摘要                                │
│ 用于什么｜何时用｜何时不用｜关键规则           │
├──────────────────────────────────────────────┤
│ 决策路径                                     │
│ 问题 → 条件 → 类型/模式 → 对应 Rule ID        │
├──────────────────────────────────────────────┤
│ 核心规则                                     │
│ Rule ID｜条件｜要求｜原因｜验证｜例外           │
├──────────────────────────────────────────────┤
│ 角色视图 Tabs                                │
│ 设计｜开发｜产品｜验收                        │
├──────────────────────────────────────────────┤
│ 参考区                                       │
│ 结构｜状态｜规格｜多端｜无障碍｜QA｜变更        │
└──────────────────────────────────────────────┘
```

## 七、站点级信息架构

站点导航不建议只按“组件分类”组织，可以增加任务型入口。

### 主导航：按设计对象

- 基础组件
- 导航
- 容器与浮层
- 输入
- 反馈
- 页面模式
- 多端与响应式
- Token

### 辅助入口：按任务

- **了解**：概念、原则、设计意图
- **判断**：组件选择、模式选择、决策树
- **设计**：结构、视觉、状态、内容
- **实现**：行为、Token、响应式、无障碍
- **验收**：QA 清单、边界场景、失败示例
- **变更**：新增、调整、废弃、迁移

### 筛选维度

- 角色：设计 / 开发 / 产品 / QA
- 平台：手机 / 折叠屏 / 平板 / 桌面 / 其他终端
- 输入方式：触控 / 键鼠 / 手写笔 / 语音 / 遥控器
- 状态：草稿 / 稳定 / 待废弃 / 已废弃
- 规范级别：必须 / 应 / 建议 / 可

这些维度应作为 frontmatter 元数据和搜索标签存在，不应通过复制目录实现。

## 八、Fumadocs 落地方式

### 8.1 使用内置能力

Fumadocs 当前已经提供：

- MDX 内容与自定义组件
- 可持久化、可共享选择状态的 Tabs
- Accordion、Steps、Cards、Inline TOC 等文档组件
- 基于 `structuredData` 的文档搜索
- 搜索 Tag Filter
- 使用 Zod 扩展 frontmatter schema，并在构建时验证
- 输出 processed Markdown
- 生成 `llms.txt`、`llms-full.txt` 和单页 Markdown 路由
- 接入 Ask AI

因此无需更换文档框架，重点是补充内容模型和自定义组件。

### 8.2 建议的 frontmatter

```yaml
---
title: 抽屉浮窗
description: 用于在保留当前上下文时承载临时或并行任务
specId: component.drawer
version: 4.0
status: stable
owners:
  - System UI Design
audiences:
  - design
  - development
  - product
  - qa
platforms:
  - phone
  - foldable
  - tablet
tags:
  - overlay
  - container
  - multitasking
lastReviewed: 2026-07-17
aliases:
  - 半屏面板
  - BottomSheet
---
```

建议通过 `source.config.ts` 的 Zod schema 强制校验 `specId`、`version`、`status`、`audiences`、`platforms` 和 `lastReviewed`。

### 8.3 建议新增的 MDX 组件

第一阶段只需要 6 个组件：

| 组件 | 用途 |
| --- | --- |
| `SpecMeta` | 显示状态、版本、平台、负责人和复核日期 |
| `DecisionSummary` | 页面顶部的快速判断摘要 |
| `DecisionTree` | 将条件判断连接到类型和规则 ID |
| `Rule` | 呈现权威原子规则 |
| `AudienceTabs` | 聚合设计、开发、产品和验收视图 |
| `AcceptanceChecklist` | 可勾选或可复制的 QA 验收项 |

已有的表格、Callout、Steps、Tabs 和 Accordion 可以继续复用，不需要将所有内容组件化。

### 8.4 搜索策略

搜索结果应尽量直接命中规则，而不只是命中页面标题。

建议索引：

- 页面标题、别名和描述
- Rule ID
- 规则正文
- 适用条件
- 规范级别
- 平台和角色标签
- Token 名称
- 旧版术语

示例查询：

- “模态抽屉背景能不能点击”
- “横屏底部抽屉”
- “抽屉 L 档高度”
- “半屏面板旧规范”

搜索结果应返回对应规则锚点，例如：

```text
/components/drawer#rule-drawer-modal-background-interaction
```

### 8.5 AI 输出策略

AI 友好不等于给页面增加一个 AI Tab，而是保证 AI 获得与人相同的权威内容。

建议：

1. 启用 `includeProcessedMarkdown`。
2. 提供 `llms.txt`，让 AI 了解文档树。
3. 提供 `llms-full.txt`，用于完整离线读取。
4. 为每篇规范提供 `.md` 或 `.mdx` Markdown 路由。
5. 在 processed Markdown 中保留 Rule ID、标题、条件、规则、验证和例外。
6. AI 回答规范问题时，要求返回规则 ID、版本和页面锚点。
7. 图片中的关键判断必须有文字等价描述。

## 九、写作规则

为了同时服务人和 AI，每条规范建议遵循以下写法：

### 推荐

- 一条规则只表达一个主要判断。
- 先写适用条件，再写要求。
- 明确使用“必须、不得、应、建议、可”。
- 将原因与规则正文分开。
- 将例外写成显式字段，不藏在备注里。
- 验收条件使用可观察行为，不使用“体验良好”等抽象表达。
- 表格单元格避免塞入多个无关规则。
- 图片必须配文字说明，并关联规则 ID。
- 旧术语作为 alias 保留，不在正文中反复混用。

### 避免

- 为每个角色复制一遍同一规则。
- 只展示尺寸图，不提供文字规格。
- 只写“建议这样做”，不写适用条件和理由。
- 把关键规则藏在 Accordion 或默认未选中的 Tab 中。
- 在一段话中同时描述默认规则、例外、历史原因和实现限制。
- AI 摘要与正式规范分开维护。

## 十、迁移建议

不需要一次性重写所有规范，可以分三阶段进行。

### 阶段 1：改造一个代表性组件

选择当前信息较完整的“抽屉浮窗”作为试点：

1. 保留现有正文。
2. 补充 frontmatter 元数据。
3. 在顶部增加 30 秒判断摘要。
4. 把 8–12 条关键规则提取为带 ID 的规则单元。
5. 将现有 QA 与关键规则建立对应关系。

目标是验证页面结构，不先开发复杂筛选系统。

### 阶段 2：建立组件与写作约束

1. 实现 6 个基础 MDX 组件。
2. 建立 frontmatter schema 校验。
3. 规定 Rule ID 命名方式。
4. 在模板中加入规则字段和迁移检查项。
5. 为搜索加入角色、平台和状态 Tag。

### 阶段 3：增强 AI 与跨文档能力

1. 生成 `llms.txt`、`llms-full.txt` 和单页 Markdown。
2. 让 Ask AI 返回 Rule ID 和来源链接。
3. 增加规则级反馈与错误报告。
4. 建立规则关系图和废弃规则跳转。
5. 根据真实搜索词优化别名和摘要。

## 十一、评估标准

新样式是否有效，可以用真实任务测试，而不是只做视觉评审。

| 任务 | 成功标准 |
| --- | --- |
| 产品判断是否使用该组件 | 30 秒内找到“何时使用 / 不使用” |
| 设计师选择正确类型 | 1 分钟内通过决策路径得出结果 |
| 开发确认一个行为 | 能命中单条规则，并找到状态、例外和验收条件 |
| QA 编写测试用例 | 关键规则都有可执行检查项 |
| AI 回答规范问题 | 返回正确规则、适用条件、版本和来源锚点 |
| 规范更新 | 修改一条事实时，不需要同步多个角色版本 |

建议在试点阶段记录：

- 查找正确规则所需时间
- 首次判断正确率
- 搜索无结果率
- AI 回答引用正确率
- 规则更新后的重复修改数量

---

## 十二、推荐的最小可行版本

如果只做一版 MVP，建议先实现以下组合：

1. 页面顶部 `SpecMeta`。
2. 一个 `DecisionSummary`。
3. 一个类型选择矩阵或决策树。
4. 8–12 条带稳定 ID 的 `Rule`。
5. 设计 / 开发 / 产品 / 验收四个角色 Tabs。
6. 保留现有规格表、状态表、QA 和更改日志。
7. 输出单页 Markdown 和 `llms-full.txt`。

这一版本已经能显著提高可读性、判断效率和 AI 可用性，同时不会破坏现有 Fumadocs 内容体系。
